## What is Docker?

**Docker** is a platform/tool that allows you to build, run, and manage applications inside **containers**. Think of Docker as the engine that powers containers.

### Docker provides:
- ✓ **Docker Engine** (the runtime for containers)
- ✓ **Docker CLI** (docker commands)
- ✓ **Docker Hub** (a registry for container images)

> **In short:** Docker is the technology that helps create and manage containers.

---

## Containers Explained

A **container** is a lightweight, isolated environment where an application runs along with everything it needs: libraries, dependencies, configuration.

It's like a virtual machine but much more efficient because it shares the host OS kernel instead of having its own.

### Key Characteristics:
| Characteristic | Description |
|----------------|-------------|
| **Portable** | Runs the same way on any system that has Docker installed |
| **Isolated** | Doesn't interfere with other apps or the host system |
| **Fast & Lightweight** | Starts in seconds because it doesn't boot a full OS |

### Simple Analogy
- **Container:** A shipping container holding your goods (the app and its dependencies)
- **Docker:** The shipping company that standardizes containers, moves them efficiently, and makes sure they fit everywhere

---

## Containers vs Virtual Machines (VMs)

| Feature | Container | Virtual Machine (VM) |
|---------|-----------|---------------------|
| **Isolation** | Process-level isolation | Full OS-level isolation |
| **Size** | Lightweight (tens or hundreds of MB) | Heavy (GBs) |
| **Startup Time** | Seconds | Minutes |
| **OS** | Shares host OS kernel | Each VM runs its own OS |
| **Resource Usage** | Very efficient (low overhead) | High overhead due to extra OS layer |
| **Portability** | Highly portable (works anywhere with Docker) | Less portable, tied to hypervisor/VM format |

### Architecture Diagram (Simplified)

**Virtual Machine:**
```
[ Hardware ]
    ↓
[ Host OS ]
    ↓
[ Hypervisor ]
    ↓
[ VM OS + App + Dependencies ]
```

**Container:**
```
[ Hardware ]
    ↓
[ Host OS + Container Engine (Docker) ]
    ↓
[ App + Dependencies ]
```

### Key Difference:
- **VMs** virtualize the hardware → run entire OS inside them
- **Containers** virtualize the OS → share the kernel, so they're much lighter

### ✅ Analogy
- **VM** = Renting a full house (everything separate: kitchen, plumbing, walls)
- **Container** = Renting a room in a shared building (you share walls and utilities but keep your room separate)

## Why Do We Need Containers?

### 1. Machine Independent
- **Without containers:** An app might run on your laptop but fail in production due to different OS versions or missing libraries
- **Container solution:** Bundle the app with all its dependencies in one portable unit → runs anywhere

### 2. Portability
Same container image runs:
- ✓ On your laptop
- ✓ On a testing server
- ✓ In the cloud (AWS, Azure, GCP)

> **Write once, run anywhere.**

### 3. Lightweight & Fast
- Containers share the host OS kernel, so:
  - Start in seconds, not minutes like VMs
  - Use less RAM and CPU → better resource utilization

### 4. Scalability
- Need more traffic handling? Just spin up more containers
- Works perfectly with microservices architecture

### 5. Isolation & Security
- Each container runs in its own isolated environment:
  - If one crashes, others keep running
  - Limits impact of security vulnerabilities

### 6. Easy CI/CD Integration
- Containers make continuous integration and deployment smooth
- Build image once, deploy the same image across environments

## Docker Architecture

Docker uses a **client-server model**. Here are the main components:

### Components of Docker Architecture

| Component | Description |
|-----------|-------------|
| **Docker Client** | CLI tool (`docker` command) that you use to interact with Docker. Sends commands to the Docker Daemon via REST API. |
| **Docker Daemon (dockerd)** | Runs on the host machine. Builds, runs, and manages containers and images. |
| **Docker Images** | Read-only templates used to create containers. |
| **Docker Containers** | Running instances of Docker images. |
| **Docker Registry** | Stores images (e.g., Docker Hub). When you run `docker pull`, images come from a registry. |

## Docker Swarm Basics

### What is Docker Swarm?

**Docker Swarm** is Docker's native clustering and orchestration solution. It allows you to group multiple Docker hosts (nodes) into a Swarm cluster and manage them as a single virtual system.

| Without Swarm | With Swarm |
|---------------|------------|
| You manage individual Docker engines manually | You manage multiple Docker hosts together |
| Manual deployment and scaling | Easy deployment and scaling across machines |

### Key Concepts

| Concept | Description |
|---------|-------------|
| **Node** | An instance of the Docker engine, which can be either a manager or a worker |
| **Manager Node** | Responsible for managing the swarm, making decisions about the cluster, and scheduling services |
| **Worker Node** | Executes the tasks assigned by the manager node |
| **Service** | Defines how to run a specific task or container in the swarm |
| **Task** | A running container managed by Docker Swarm |
| **Swarm** | A cluster of Docker nodes that run in swarm mode |
| **Overlay Network** | Enables communication between containers deployed across different nodes |

### Architecture Components

1. **Control Plane:** Managed by manager nodes, responsible for:
   - Maintaining the state of the swarm
   - Scheduling tasks on worker nodes
   - Providing an API for users
   - Handling cluster management tasks

2. **Data Plane:** Composed of worker nodes, responsible for:
   - Running the containers (tasks) defined by the services
   - Pulling images from the Docker registry
   - Reporting status back to manager nodes

### Features

- ✓ **High Availability:** Multiple manager nodes prevent single points of failure
- ✓ **Load Balancing:** Automatically distributes requests among service replicas
- ✓ **Scaling:** Easily scale services up or down by changing the number of replicas
- ✓ **Rolling Updates:** Update services without downtime
- ✓ **Secrets Management:** Securely manage sensitive information

## Installation Guide

### OS Requirements for Docker Engine

You need the 64-bit version of one of these Ubuntu versions:
- Ubuntu Oracular 24.10
- Ubuntu Noble 24.04 (LTS)
- Ubuntu Jammy 22.04 (LTS)

### Step 1: Remove Conflicting Packages

```bash
$ for pkg in docker.io docker-doc docker-compose docker-compose-v2 podman-docker containerd runc; do sudo apt-get remove $pkg; done
```

### Step 2: Install Using the apt Repository

```bash
# Add Docker's official GPG key
sudo apt-get update
sudo apt-get install ca-certificates curl
sudo install -m 0755 -d /etc/apt/keyrings
sudo curl -fsSL https://download.docker.com/linux/ubuntu/gpg -o /etc/apt/keyrings/docker.asc
sudo chmod a+r /etc/apt/keyrings/docker.asc

# Add the repository to Apt sources
echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.asc] https://download.docker.com/linux/ubuntu \
  $(. /etc/os-release && echo "${UBUNTU_CODENAME:-$VERSION_CODENAME}") stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null
sudo apt-get update

# Install Docker packages
sudo apt-get install docker-ce docker-ce-cli containerd.io
sudo systemctl enable docker
sudo systemctl start docker
```

### Step 3: Execute Docker Command Without Sudo (Optional)

```bash
# Add your user to the docker group
sudo usermod -aG docker ${USER}

# Refresh group assignment
su - ${USER}

# Or add a specific user
sudo usermod -aG docker username
```

### Step 4: Verify Installation

```bash
sudo docker run hello-world

# Check Docker status
echo "🔍 Checking Docker status..."
if systemctl is-active --quiet docker; then
    echo "🎉 Docker is installed and running!"
    docker --version
else
    echo "❌ Docker installation failed or service is not running."
    exit 1
fi
```

### Internet/Proxy Configuration (If Needed)

```bash
# Configure DNS
sudo echo '{ "dns": ["8.8.8.8","1.1.1.1"] }' | sudo tee /etc/docker/daemon.json

# Configure proxy (if behind corporate proxy)
sudo mkdir -p /etc/systemd/system/docker.service.d
sudo vi /etc/systemd/system/docker.service.d/proxy.conf
```

Add to `proxy.conf`:
```ini
[Service]
Environment="HTTP_PROXY=http://your-proxy:8080"
Environment="HTTPS_PROXY=http://your-proxy:8080"
Environment="NO_PROXY=localhost,127.0.0.1"
```

Then reload:
```bash
sudo systemctl daemon-reload
sudo systemctl restart docker
```

## Docker Swarm Setup

### Prerequisites & Ports

Open these ports between nodes (manager ↔ workers):

| Port | Protocol | Purpose |
|------|----------|---------|
| 2377 | TCP | Cluster management (join) |
| 7946 | TCP/UDP | Node gossip |
| 4789 | UDP | Overlay networking (VXLAN) |

**Example with UFW:**
```bash
sudo ufw allow 2377/tcp
sudo ufw allow 7946
sudo ufw allow 4789/udp
```

### Step 1: Initialize the First Manager

```bash
sudo docker swarm init --advertise-addr <MANAGER-IP>
```

**Example output:**
```
Swarm initialized: current node (dxn1zf6l61qsb1josjja83ngz) is now a manager.

To add a worker to this swarm, run the following command:
    docker swarm join \
    --token SWMTKN-1-49nj1cmql0jkz5s954yi3oex3nedyz0fb0xx14ie39trti4wxv-8vxv8rssmk743ojnwacrr2e7c \
    192.168.99.100:2377
```

### Step 2: Add Nodes to the Swarm

**For a 3-node cluster (all nodes as managers and workers):**

```bash
# On node 170 (manager)
docker swarm init --advertise-addr 172.21.1.170

# On node 171
docker swarm join --token <TOKEN> 172.21.1.170:2377

# On node 172
docker swarm join --token <TOKEN> 172.21.1.170:2377
```

**On each worker node:**
```bash
sudo docker swarm join --token <WORKER-TOKEN> <MANAGER-IP>:2377
```

**To join a node as a manager:**
```bash
sudo docker swarm join --token <MANAGER-TOKEN> <MANAGER-IP>:2377
```

### Useful Swarm Commands

```bash
# View swarm state
docker info

# View nodes in the swarm
docker node ls

# Get join tokens
docker swarm join-token worker
docker swarm join-token manager

# Get only token string
docker swarm join-token worker -q

# Drain a node for maintenance
docker node update --availability drain <NODE-ID>

# Reactivate a node after maintenance
docker node update --availability active <NODE-ID>

# Remove a node from the cluster (on manager)
docker node rm <NODE-ID>

# Leave the swarm (on the node itself)
docker swarm leave

# Force leave for a manager
docker swarm leave --force
```

## Portainer Installation

Portainer provides a web-based UI for managing Docker and Swarm clusters.

### Install Portainer CE with Docker Swarm

**Step 1: Download the stack YML manifest**
```bash
curl -L https://downloads.portainer.io/ce-lts/portainer-agent-stack.yml -o portainer-agent-stack.yml
```

**Step 2: Deploy the stack**
```bash
docker stack deploy -c portainer-agent-stack.yml portainer
```

### Example Stack YML File

```yaml
version: '3.2'

services:
  agent:
    image: portainer/agent:lts
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock
      - /var/lib/docker/volumes:/var/lib/docker/volumes
    networks:
      - agent_network
    deploy:
      mode: global
      placement:
        constraints: [node.platform.os == linux]

  portainer:
    image: portainer/portainer-ce:lts
    command: -H tcp://tasks.agent:9001 --tlsskipverify
    ports:
      - "9000:9000"
      - "8000:8000"
    volumes:
      - portainer_data:/data
    networks:
      - agent_network
    deploy:
      mode: replicated
      replicas: 1
      placement:
        constraints: [node.role == manager]

networks:
  agent_network:
    driver: overlay
    attachable: true

volumes:
  portainer_data:
```

**Verify installation:**
```bash
docker ps
```

Access Portainer at `http://<manager-ip>:9000`

## NFS Setup

### Install NFS Server

**Step 1: Install NFS Kernel Server**
```bash
apt update
apt install nfs-kernel-server

# If the same server needs to act as client
apt install nfs-common
```

**Step 2: Configure Shared Directory**
```bash
# Create shared directory
mkdir -p /mnt/nfs_share

# Change ownership to public
chown nobody:nogroup /mnt/nfs_share

# Set permissions
chmod 777 /mnt/nfs_share
```

**Step 3: Edit NFS Export File**
```bash
vi /etc/exports
```

Add this line:
```
/mnt/nfs_share 172.21.1.0/24(rw,sync,no_subtree_check)
```

**Step 4: Export and Restart**
```bash
exportfs -a
systemctl restart nfs-kernel-server
```

### Install NFS Client

**Step 1: Install NFS Common**
```bash
apt install nfs-common -y
```

**Step 2: Create Mount Point**
```bash
mkdir -p /mnt/nfsdir_client
```

**Step 3: Mount Shared Directory**
```bash
mount 172.21.1.170:/mnt/nfs_share /mnt/nfsdir_client

# Verify mount
df -h
ls -lrt /mnt/nfs_share/
```

**Step 4: Test the NFS Share**
```bash
# On server
touch /mnt/nfs_share/test.yaml

# On client
ls -lrt /mnt/nfsdir_client/
```

**Step 5: Mount on OS Boot**

Add to `/etc/fstab`:
```bash
vi /etc/fstab
```

Add this line:
```
172.21.1.170:/mnt/nfs_share /mnt/nfsdir_client nfs defaults 0 0
```

## Local Docker Registry

### Why Use a Local Registry?

- **Speed:** Eliminates latencies from downloading images from the internet
- **Control:** Full control over who accesses your images
- **Security:** Keeps sensitive images off public networks
- **Efficiency:** Integrates with internal CI/CD workflows

### Setup Local Registry

**Step 1: Run the Registry Container**
```bash
docker run -d -p 5000:5000 --name local-registry registry:latest

# Verify status
docker container ls
```

**Step 2: Pull, Tag, and Push an Image**
```bash
# Pull an image
docker pull nginx:latest

# Tag for local registry
docker tag nginx:latest localhost:5000/nginx:latest

# Push to local registry
docker push localhost:5000/nginx:latest
```

**Step 3: Verify**
```bash
# Check inside container
docker exec -it local-registry sh
# cd /var/lib/registry/docker/registry/v2/repositories
# ls
nginx

# Pull from registry
docker pull localhost:5000/nginx:latest
```

### Configure Insecure Registry (If Needed)

If clients refuse to pull because registry is "insecure", add to `/etc/docker/daemon.json`:

```json
{
  "insecure-registries": ["172.21.1.170:5000"]
}
```

Then restart:
```bash
systemctl restart docker
```

### List Images and Tags

```bash
# List repositories
curl http://localhost:5000/v2/_catalog

# List tags for a repository
curl http://localhost:5000/v2/harbortest/nginx/tags/list

# Better output with jq
curl -s http://localhost:5000/v2/_catalog | jq -r '.repositories[]'
```

### Run Container from Local Registry
```bash
docker run -d --name test_registry_02_n2 172.21.1.170:5000/nginx:latest
```

## Harbor Registry

Harbor is an enterprise-class Docker registry with advanced features like vulnerability scanning, replication, and role-based access control.

### Install Harbor on Ubuntu 24.04

**Step 1: Download Harbor**
```bash
# Download the latest offline installer
curl -s https://api.github.com/repos/goharbor/harbor/releases/latest | grep browser_download_url | cut -d '"' -f 4 | grep '\.tgz$' | wget -i -

# Extract
tar xvf harbor-offline-installer-v2.13.2.tgz
cd harbor
cp harbor.yml.tmpl harbor.yml
```

**Step 2: Configure harbor.yml**
```yaml
# The IP address or hostname to access admin UI and registry service
# DO NOT use localhost or 127.0.0.1
hostname: 172.21.1.171

# http related config
http:
  port: 80
```

**Step 3: Install Harbor with Trivy (Vulnerability Scanner)**
```bash
./install.sh --with-trivy

# Verify services
docker compose ps

# Check listening ports
lsof -i -P -n | grep LISTEN
```

**Step 4: Validate Installation**
```bash
# List repositories
curl -vk -u admin:Harbor12345 http://172.21.1.171/v2/_catalog

# Check service status
cd ~/harbor
docker compose ps
```

### Harbor Management Commands

```bash
# Stop Harbor
docker compose down

# Start Harbor
docker compose up -d

# Check logs
docker compose logs
```

### Push Images to Harbor

**Step 1: Login to Harbor**
```bash
docker login 172.21.1.171:80
# Username: admin
# Password: Harbor12345
```

**Step 2: Pull or Build an Image**
```bash
docker pull nginx:latest
```

**Step 3: Tag for Harbor**
```bash
docker tag nginx:latest 172.21.1.171:80/myproject/nginx:latest
```

**Step 4: Push to Harbor**
```bash
docker push 172.21.1.171:80/myproject/nginx:latest
```

**Step 5: Verify in Harbor UI**
- Open `http://172.21.1.171`
- Navigate to Projects → myproject → Repositories
- You should see `nginx` with the `latest` tag

## GitLab Installation

### Install GitLab CE on Ubuntu Server 24.04

**Step 1: Update and Install Dependencies**
```bash
sudo apt update
sudo apt install ca-certificates curl openssh-server postfix tzdata perl
```

**Step 2: Install GitLab CE**
```bash
cd /tmp
curl -LO https://packages.gitlab.com/install/repositories/gitlab/gitlab-ce/script.deb.sh
sudo bash /tmp/script.deb.sh
sudo apt install gitlab-ce
```

**Step 3: Configure GitLab**
```bash
sudo vi /etc/gitlab/gitlab.rb
```

Set the external URL:
```ruby
external_url "http://172.21.1.172"
```

**Step 4: Reconfigure GitLab**
```bash
sudo gitlab-ctl reconfigure
```

### Reset Root Password

```bash
sudo gitlab-rails console -e production
```

In the Rails console:
```ruby
user = User.find_by(username: 'root')
user.password = 'NewStr0ngP@ssw0rd'
user.password_confirmation = 'NewStr0ngP@ssw0rd'
user.save!
exit
```

### Configure Let's Encrypt SSL

Edit `/etc/gitlab/gitlab.rb`:

```ruby
external_url 'https://172.21.1.172'

# Enable Let's Encrypt
letsencrypt['enable'] = true
letsencrypt['contact_emails'] = ['your-email@example.com']

# Enable auto-renew
letsencrypt['auto_renew'] = true

# Set renewal schedule (every 7th day at 12:30)
letsencrypt['auto_renew_hour'] = "12"
letsencrypt['auto_renew_minute'] = "30"
letsencrypt['auto_renew_day_of_month'] = "*/7"
```

Apply changes:
```bash
sudo gitlab-ctl reconfigure
```

## Best Practices

1. **Use .dockerignore** files to exclude unnecessary files from builds
2. **Keep images small** - use alpine or slim variants when possible
3. **Use specific tags** instead of `latest` for production
4. **Run containers as non-root users** for security
5. **Regularly update base images** for security patches
6. **Use multi-stage builds** to reduce final image size
7. **Label your images** with metadata (version, maintainer, etc.)
8. **Scan images for vulnerabilities** using tools like Trivy or Docker Scout
