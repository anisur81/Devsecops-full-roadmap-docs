# Kubernetes Logging Architecture

## Overview

The native functionality provided by a container engine or runtime is usually not enough for a complete logging solution. In a cluster, logs should have a separate storage and lifecycle independent of nodes, pods, or containers. This concept is called **cluster-level logging**.

> **Note:** Cluster-level logging architectures require a separate backend to store, analyze, and query logs. Kubernetes does not provide a native storage solution for log data.

---

## Pod and Container Logs

Kubernetes captures logs from each container in a running Pod.

### Example: Basic Pod with Logging

This example uses a manifest for a Pod with a container that writes text to the standard output stream once per second:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: counter
spec:
  containers:
  - name: count
    image: busybox:1.28
    args: [/bin/sh, -c,
            'i=0; while true; do echo "$i: $(date)"; i=$((i+1)); sleep 1; done']
```

#### Running the Pod

```bash
kubectl apply -f https://k8s.io/examples/debug/counter-pod.yaml
```

**Output:**
```
pod/counter created
```

#### Fetching Logs

```bash
kubectl logs counter
```

**Output:**
```
0: Fri Apr  1 11:42:23 UTC 2022
1: Fri Apr  1 11:42:24 UTC 2022
2: Fri Apr  1 11:42:25 UTC 2022
```

---

### Viewing Previous Container Logs

Use `kubectl logs --previous` to retrieve logs from a previous instantiation of a container:

```bash
kubectl logs counter --previous
```

### Multiple Containers

If your pod has multiple containers, specify which container's logs you want to access by appending a container name with the `-c` flag:

```bash
kubectl logs counter -c count
```

## Container Log Streams

Kubernetes 1.32 allows access to these log streams directly via the Pod API. You can fetch a specific stream by specifying the stream name (either `Stdout` or `Stderr`) using the stream query string. You must have access to read the log subresource of that Pod.

### Example: Pod Writing to Both Output Streams

Create a Pod that periodically writes text to both the standard output and error stream:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: counter-err
spec:
  containers:
  - name: count
    image: busybox:1.28
    args: [/bin/sh, -c,
            'i=0; while true; do echo "$i: $(date)"; echo "$i: err" >&2 ; i=$((i+1)); sleep 1; done']
```

#### Running the Pod

```bash
kubectl apply -f https://k8s.io/examples/debug/counter-pod-err.yaml
```

#### Fetching Only stderr Log Stream

```bash
kubectl get --raw "/api/v1/namespaces/default/pods/counter-err/log?stream=Stderr"
```

## Cluster-Level Logging Architectures

While Kubernetes does not provide a native solution for cluster-level logging, there are several common approaches to consider:

### Available Options

1. **Node-level logging agent** - Runs on every node
2. **Sidecar container** - Dedicated logging container in an application pod
3. **Direct push** - Push logs directly to a backend from within an application

---

### Using a Sidecar Container with the Logging Agent

You can use a sidecar container in one of the following ways:

1. The sidecar container streams application logs to its own `stdout`
2. The sidecar container runs a logging agent configured to pick up logs from an application container

#### Example: Sidecar with Multiple Log Files

A pod runs a single container that writes to two different log files using two different formats:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: counter
spec:
  containers:
  - name: count
    image: busybox:1.28
    args:
    - /bin/sh
    - -c
    - >
      i=0;
      while true;
      do
        echo "$i: $(date)" >> /var/log/1.log;
        echo "$(date) INFO $i" >> /var/log/2.log;
        i=$((i+1));
        sleep 1;
      done      
    volumeMounts:
    - name: varlog
      mountPath: /var/log
  volumes:
  - name: varlog
    emptyDir: {}
```

## Summary

| Approach | Description | Best For |
|----------|-------------|----------|
| **Node-level Agent** | Agent runs on each node collecting logs from all pods | Large clusters, centralized logging |
| **Sidecar Container** | Dedicated container per pod for log processing | Specialized log formatting requirements |
| **Direct Push** | Application pushes logs directly to backend | Applications with built-in logging support |
