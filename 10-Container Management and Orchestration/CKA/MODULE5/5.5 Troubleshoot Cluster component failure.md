# Troubleshooting Clusters - A Comprehensive Guide

## Overview

This guide focuses on debugging common cluster issues in Kubernetes. Before proceeding, ensure you have ruled out application-specific problems by consulting the [application troubleshooting guide](link-to-app-troubleshooting).

---

## Prerequisites

- `kubectl` configured and working
- Access to cluster nodes (SSH or equivalent)
- Appropriate permissions to view logs and cluster resources

## 2. Node Troubleshooting

### Diagnosing a Non-Ready Node

When a node shows `NotReady` status:

```bash
# Get detailed node information
kubectl describe node <node-name>

# Get node YAML representation
kubectl get node <node-name> -o yaml
```

**Common Node Conditions:**

| Condition | Status | Meaning |
|-----------|--------|---------|
| Ready | True/False/Unknown | Node is healthy/unhealthy/status unknown |
| MemoryPressure | True/False | Node has memory pressure |
| DiskPressure | True/False | Node has disk pressure |
| PIDPressure | True/False | Node has PID pressure |
| NetworkUnavailable | True/False | Network not configured |

### Example: Down/Unreachable Node

```yaml
# Output from kubectl describe node
Name:               kube-worker-1
Status:             NotReady

# Notice the taints and conditions
Taints:
  node.kubernetes.io/unreachable:NoExecute
  node.kubernetes.io/unreachable:NoSchedule

Conditions:
  Ready                Unknown   Kubelet stopped posting node status.
  MemoryPressure       Unknown   Kubelet stopped posting node status.
  DiskPressure         Unknown   Kubelet stopped posting node status.
```

**Implications:**
- Pods on the node will be evicted after ~5 minutes
- New pods won't be scheduled on this node
- Services relying on these pods will be affected

## 3. Log Analysis

### Systemd-based Systems

```bash
# View kubelet logs
journalctl -u kubelet -f

# View kubelet logs with timestamps
journalctl -u kubelet --since "2024-01-01 10:00:00" --until "2024-01-01 11:00:00"

# View all kubernetes component logs
journalctl -u kubelet -u kube-proxy -u docker -f
```

### Traditional Log File Locations

#### Control Plane Nodes

| Component | Log Path | Purpose |
|-----------|----------|---------|
| API Server | `/var/log/kube-apiserver.log` | Serves the Kubernetes API |
| Scheduler | `/var/log/kube-scheduler.log` | Makes scheduling decisions |
| Controller Manager | `/var/log/kube-controller-manager.log` | Manages built-in controllers |

#### Worker Nodes

| Component | Log Path | Purpose |
|-----------|----------|---------|
| Kubelet | `/var/log/kubelet.log` | Manages containers on node |
| Kube-proxy | `/var/log/kube-proxy.log` | Handles service networking |

### Using Container Runtime Logs

```bash
# Docker runtime
docker ps -a
docker logs <container-id>

# Containerd
crictl ps -a
crictl logs <container-id>
```

## 4. Cluster Failure Modes

### Contributing Causes

| Cause | Impact |
|-------|--------|
| VM shutdown | Node becomes unavailable |
| Network partition | Split-brain scenarios |
| Kubernetes component crashes | API unavailable, scheduling stops |
| Storage failure | Data loss, persistent volumes unavailable |
| Operator error | Misconfiguration, accidental deletions |

### Common Scenarios

#### 1. API Server Failure

**Symptoms:**
- `kubectl` commands fail with connection refused
- Existing pods and services continue running
- Cannot create/update/delete resources

**Causes:**
- VM shutdown
- API server crash
- etcd backup storage lost

**Mitigation:**
```bash
# Check API server status
kubectl get pod -n kube-system -l component=kube-apiserver

# Check API server logs
kubectl logs -n kube-system kube-apiserver-<pod-id>

# Restart API server if needed
kubectl delete pod -n kube-system kube-apiserver-<pod-id>
```

#### 2. etcd Failure

**Symptoms:**
- API server fails to start
- Cluster state inconsistent
- `kubectl` commands fail or timeout

**Mitigation:**
```bash
# Backup etcd
ETCDCTL_API=3 etcdctl snapshot save snapshot.db

# Restore from backup
ETCDCTL_API=3 etcdctl snapshot restore snapshot.db
```

#### 3. Network Partition

**Symptoms:**
- Nodes split into partitions
- Some nodes show `NotReady`
- Services become partially unavailable

**Mitigation:**
```bash
# Check network plugin pods
kubectl get pods -n kube-system | grep -E "calico|flannel|weave"

# Check network connectivity
kubectl run test-pod --image=busybox --rm -it -- nslookup kubernetes
```

## 5. Mitigation Strategies

### High-Availability Setup

**Control Plane:**
```yaml
# Example: High-availability API Server configuration
apiVersion: v1
kind: Pod
metadata:
  name: kube-apiserver
spec:
  replicas: 3
  containers:
  - name: kube-apiserver
    args:
    - --etcd-servers=https://etcd-1:2379,https://etcd-2:2379,https://etcd-3:2379
```

### Best Practices

| Action | Purpose |
|--------|---------|
| Use IaaS auto-restart features | Recover from VM crashes |
| Use reliable storage (GCE PD, AWS EBS) | Prevent data loss |
| Implement HA configuration | Tolerate component failures |
| Regular snapshots of etcd/state | Enable disaster recovery |
| Use ReplicaSets/ReplicationControllers | Ensure pod availability |
| Design applications for restart tolerance | Handle unexpected failures |

### Node Recovery

```bash
# Drain node for maintenance
kubectl drain <node-name> --ignore-daemonsets

# Repair/restart node
systemctl restart kubelet

# Uncordon node when ready
kubectl uncordon <node-name>
```

## 6. Debugging Workflow

### Step-by-Step Approach

```
1. Identify the Problem
   ├── kubectl get nodes
   ├── kubectl get pods --all-namespaces
   └── kubectl get events --all-namespaces

2. Gather Details
   ├── kubectl describe node/node/pod
   ├── Check logs
   └── Check metrics (if available)

3. Isolate the Issue
   ├── Is it node-level? → Check kubelet
   ├── Is it network-level? → Check network plugin
   └── Is it control plane? → Check API server, etcd

4. Attempt Recovery
   ├── Restart components
   ├── Recreate resources
   └── Apply fixes or patches

5. Verify Resolution
   ├── Check status again
   ├── Test functionality
   └── Monitor for recurrence
```

## 7. Common Commands Reference

### Quick Diagnostics

```bash
# Cluster health
kubectl cluster-info
kubectl get componentstatuses

# Node status
kubectl get nodes -o wide
kubectl describe nodes

# Pod status
kubectl get pods --all-namespaces
kubectl get pods -o wide --all-namespaces

# Events
kubectl get events --all-namespaces --sort-by='.lastTimestamp'

# Resource usage
kubectl top nodes
kubectl top pods --all-namespaces
```

### Advanced Debugging

```bash
# Check kubelet health
curl -k https://<node-ip>:10250/healthz

# Check API server health
curl -k https://<api-server-ip>:6443/healthz

# View kubelet configuration
kubectl get cm -n kube-system kubelet-config -o yaml

# Check node conditions
kubectl get nodes -o custom-columns=NAME:.metadata.name,READY:.status.conditions[?(@.type=="Ready")].status,MEMORY:.status.conditions[?(@.type=="MemoryPressure")].status
```

## 8. Security Considerations

### When Troubleshooting

```bash
# Always use secure connections
kubectl --server=https://cluster.example.com:6443

# Verify certificates
openssl s_client -connect api-server:6443

# Check audit logs
kubectl logs -n kube-system kube-apiserver-<pod-id> | grep "audit"
```

### Troubleshooting RBAC Issues

```bash
# Check permissions
kubectl auth can-i get pods --as=system:serviceaccount:default:my-sa

# View roles and bindings
kubectl get clusterroles,clusterrolebindings -o yaml
kubectl get roles,rolebindings --all-namespaces -o yaml
```
