# Debugging Pods in Kubernetes

## Initial Pod Inspection

The first step in debugging a Pod is examining its current state and recent events:

```bash
kubectl describe pods ${POD_NAME}
```

**Key information to check:**
- Container states (Running, Waiting, Terminated)
- Recent restart counts
- Events that occurred during pod lifecycle

---

## Pod Stuck in "Pending" State

### Understanding Pending State
A Pod in `Pending` status means it has been accepted by the Kubernetes cluster, but one or more containers has not been set up and made ready to run.

### Common Causes and Solutions

#### 1. Insufficient Resources
**Problem:** Not enough CPU, memory, or other resources available on any node.

**Solutions:**
- Check resource requests and limits in your pod specification
- If Cluster Autoscaling is enabled, it will automatically increase node count
- Scale down other workloads to free resources
- Add more nodes to your cluster manually

**Diagnostic Commands:**
```bash
# Check node resource usage
kubectl top nodes

# Check resource requests vs available capacity
kubectl describe nodes
```

#### 2. Insufficient HostPorts
**Problem:** The pod requires specific hostPorts that are already in use.

**Solution:** Verify hostPort assignments and either free up ports or use different ports.

#### 3. Scheduling Issues
**Problem:** Pod cannot be scheduled due to node selectors, taints, or affinity rules.

**Solutions:**
- Verify node selectors match available nodes
- Check for taints and tolerations configuration
- Review pod affinity and anti-affinity rules

**Diagnostic Commands:**
```bash
# Check node labels and taints
kubectl describe nodes

# Check pod scheduling details
kubectl get events --field-selector involvedObject.name=${POD_NAME}
```

### Validation Steps

**Validate pod configuration:**
```bash
# Validate your manifest before applying
kubectl apply --validate -f mypod.yaml

# Check for misconfigurations in the pod spec
kubectl describe pod ${POD_NAME}
```

**Review pod details:**
```bash
kubectl describe pod ${POD_NAME}
```

**Check for specific issues:**
- Is `status` equal to `RUNNING`?
- Are there enough resources to schedule the pod?
- Are there enough `hostPorts` remaining to schedule the pod?
- Are node selectors properly configured?

## Pod Stuck in "Terminating" State

### Understanding Terminating State
A Pod stuck in `Terminating` state means a deletion has been issued, but the control plane cannot delete the Pod object.

### Common Causes and Solutions

#### 1. Finalizer and Admission Webhook Issues
**Problem:** The pod has a finalizer that can't be removed, often due to an admission webhook.

**Diagnosis:**
```bash
# Check for validating webhook configurations
kubectl get validatingwebhookconfigurations

# Check for mutating webhook configurations
kubectl get mutatingwebhookconfigurations

# Look for webhooks targeting UPDATE operations on pods
```

#### 2. Webhook Provider Issues
**Solutions:**
- Ensure you're using the latest version of the webhook provider
- Consider disabling the webhook for `UPDATE` operations temporarily
- Report issues to the corresponding provider
- Force delete the pod (as a last resort)

**Force deletion:**
```bash
# Remove finalizers and delete immediately
kubectl delete pod ${POD_NAME} --grace-period=0 --force

# Alternative: Patch to remove finalizers
kubectl patch pod ${POD_NAME} -p '{"metadata":{"finalizers":null}}'
```

## Pod in "Waiting" State

### Understanding Waiting State
A container in `Waiting` state is still performing required operations to complete startup.

### Debugging Steps

**1. Inspect container details:**
```bash
kubectl describe pod ${POD_NAME}
```

**2. Check for common issues:**
- Image pull errors (incorrect image name, registry authentication)
- Volume mount issues
- Configuration errors
- Resource constraints

**3. View logs:**
```bash
# Check logs from the container
kubectl logs ${POD_NAME}

# Check logs from a specific container (if multiple)
kubectl logs ${POD_NAME} -c ${CONTAINER_NAME}
```

## Additional Debugging Commands

### Event Monitoring
```bash
# Watch events for the specific pod
kubectl get events --field-selector involvedObject.name=${POD_NAME}

# Watch all events in the namespace
kubectl get events --watch
```

### Container Inspection
```bash
# Get detailed pod information
kubectl get pod ${POD_NAME} -o yaml

# Check container logs with timestamps
kubectl logs ${POD_NAME} --timestamps

# Check previous container logs (if restarted)
kubectl logs ${POD_NAME} --previous
```

### Interactive Debugging
```bash
# Execute a command in the container
kubectl exec -it ${POD_NAME} -- /bin/bash

# Execute in a specific container
kubectl exec -it ${POD_NAME} -c ${CONTAINER_NAME} -- /bin/sh

# Port forward to access container services
kubectl port-forward ${POD_NAME} 8080:80
```

## Best Practices

1. **Always check resource requests and limits** - Ensure they're realistic for your workload
2. **Monitor cluster health** - Regularly check node status and capacity
3. **Use meaningful labels** - Helps with debugging and organization
4. **Implement proper logging** - Configure appropriate log levels and aggregation
5. **Keep webhooks updated** - Regularly update third-party admission webhooks
6. **Test configurations** - Validate YAML before applying to production
7. **Use readiness probes** - Ensure containers are ready to serve traffic
8. **Review events regularly** - Proactively identify potential issues

## Troubleshooting Checklist

- [ ] Check pod status: `kubectl get pods`
- [ ] Describe pod: `kubectl describe pod ${POD_NAME}`
- [ ] Check events: `kubectl get events`
- [ ] Verify node resources: `kubectl top nodes`
- [ ] Check container logs: `kubectl logs ${POD_NAME}`
- [ ] Validate pod configuration: `kubectl apply --validate -f mypod.yaml`
- [ ] Check for webhook configurations (if terminating)
- [ ] Verify network policies (if connection issues)
- [ ] Confirm storage class availability (if using PVCs)
- [ ] Check container image availability and registry access
