# Kubernetes PersistentVolume Exercise Guide

## Overview
This guide walks through creating a PersistentVolume (PV) with `hostPath` storage, binding it to a PersistentVolumeClaim (PVC), and using it in a Pod. This is useful for development and testing on single-node clusters.

---

## Step 1: Create the PersistentVolume

### PV Configuration (`pv-volume.yaml`)
```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: task-pv-volume
  labels:
    type: local
spec:
  storageClassName: manual
  capacity:
    storage: 10Gi
  accessModes:
    - ReadWriteOnce
  hostPath:
    path: "/mnt/data"
```

### Apply the PV
```bash
kubectl apply -f https://k8s.io/examples/pods/storage/pv-volume.yaml
```

### Verify PV Creation
```bash
kubectl get pv task-pv-volume
```

**Expected Output:**
```
NAME             CAPACITY   ACCESSMODES   RECLAIMPOLICY   STATUS      CLAIM     STORAGECLASS   REASON    AGE
task-pv-volume   10Gi       RWO           Retain          Available             manual                   4s
```

> **Note:** Status is `Available` - not yet bound to any PVC.

---

## Step 2: Create a PersistentVolumeClaim

### PVC Configuration (`pv-claim.yaml`)
```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: task-pv-claim
spec:
  storageClassName: manual
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 3Gi
```

### Apply the PVC
```bash
kubectl apply -f https://k8s.io/examples/pods/storage/pv-claim.yaml
```

### Verify Binding
```bash
kubectl get pv task-pv-volume
```

**Expected Output:**
```
NAME             CAPACITY   ACCESSMODES   RECLAIMPOLICY   STATUS    CLAIM                   STORAGECLASS   REASON    AGE
task-pv-volume   10Gi       RWO           Retain          Bound     default/task-pv-claim   manual                   2m
```

### Check PVC Status
```bash
kubectl get pvc task-pv-claim
```

**Expected Output:**
```
NAME            STATUS    VOLUME           CAPACITY   ACCESSMODES   STORAGECLASS   AGE
task-pv-claim   Bound     task-pv-volume   10Gi       RWO           manual         30s
```

## Step 3: Create a Pod Using the PVC

### Pod Configuration (`pv-pod.yaml`)
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: task-pv-pod
spec:
  volumes:
    - name: task-pv-storage
      persistentVolumeClaim:
        claimName: task-pv-claim
  containers:
    - name: task-pv-container
      image: nginx
      ports:
        - containerPort: 80
          name: "http-server"
      volumeMounts:
        - mountPath: "/usr/share/nginx/html"
          name: task-pv-storage
```

### Apply the Pod
```bash
kubectl apply -f https://k8s.io/examples/pods/storage/pv-pod.yaml
```

### Verify Pod is Running
```bash
kubectl get pod task-pv-pod
```

### Test the Mounted Storage
```bash
# Get a shell in the container
kubectl exec -it task-pv-pod -- /bin/bash

# Inside the container, install curl and test
apt update
apt install curl -y
curl http://localhost/
```

**Expected Output:**
```
Hello from Kubernetes storage
```

## Advanced: Multiple Mounts with Same PVC

### Example: Mounting Same PVC in Multiple Locations

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: test
spec:
  containers:
    - name: test
      image: nginx
      volumeMounts:
        # Mount for static website content
        - name: config
          mountPath: /usr/share/nginx/html
          subPath: html
        # Mount for nginx configuration
        - name: config
          mountPath: /etc/nginx/nginx.conf
          subPath: nginx.conf
  volumes:
    - name: config
      persistentVolumeClaim:
        claimName: test-nfs-claim
```

### Key Concepts:

| Component | Description |
|-----------|-------------|
| **PV** | Physical storage resource (10Gi hostPath) |
| **PVC** | Storage request (3Gi minimum) |
| **Pod** | Uses PVC as volume mount |
| **subPath** | Mounts specific subdirectory/file from volume |
| **hostPath** | Node directory path (for development only) |

### Important Notes:

1. **StorageClass**: Must match between PV and PVC (`manual`)
2. **AccessModes**: `ReadWriteOnce` means single node read/write
3. **Capacity**: PVC requests 3Gi, PV provides 10Gi
4. **hostPath**: Only for development - not suitable for production
5. **subPath**: Allows mounting specific files/directories from the same volume

## Troubleshooting

### Common Issues:

```bash
# Check PV status
kubectl describe pv task-pv-volume

# Check PVC status
kubectl describe pvc task-pv-claim

# Check Pod events
kubectl describe pod task-pv-pod

# Check logs
kubectl logs task-pv-pod

# Check node directory exists (for hostPath)
ls -la /mnt/data
```

### Cleanup:
```bash
kubectl delete pod task-pv-pod
kubectl delete pvc task-pv-claim
kubectl delete pv task-pv-volume
```

## Summary

This exercise demonstrates:

✅ Creating a `hostPath` PersistentVolume  
✅ Binding a PersistentVolumeClaim to the PV  
✅ Using the PVC in a Pod  
✅ Mounting the same PVC in multiple locations with `subPath`  

**Best Practice:** Use `hostPath` only for development. Production environments should use network storage solutions like NFS, CSI drivers, or cloud provider storage.
