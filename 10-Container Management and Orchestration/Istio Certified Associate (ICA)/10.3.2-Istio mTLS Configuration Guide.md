# Istio mTLS Configuration Guide

## Overview

In Istio, mTLS (mutual TLS) has three main levels:

| Mode | Description | Use Case |
|------|-------------|----------|
| **DISABLE** | Plain traffic only | Development/testing without security |
| **PERMISSIVE** | Accept both plaintext + mTLS | Migration phase from plain to encrypted traffic |
| **STRICT** | Only mTLS allowed | Production-ready recommended final state |

---

## 1. Enable Mesh-wide STRICT mTLS

This configuration enforces mTLS across the entire mesh for the `observability` namespace.

### PeerAuthentication Configuration

**File:** `mesh-mtls.yaml`

```yaml
apiVersion: security.istio.io/v1
kind: PeerAuthentication
metadata:
  name: default
  namespace: observability
spec:
  mtls:
    mode: STRICT
```

**Apply the configuration:**
```bash
kubectl apply -f mesh-mtls.yaml
```

> **Note:** For mesh-wide strict mTLS, apply this in the `istio-system` namespace.

---

## 2. DestinationRule for mTLS

DestinationRules define how the client side should handle TLS communication.

**File:** `dr.yaml`

```yaml
apiVersion: networking.istio.io/v1
kind: DestinationRule
metadata:
  name: observability-mtls
  namespace: observability
spec:
  host: "*.observability.svc.cluster.local"
  trafficPolicy:
    tls:
      mode: ISTIO_MUTUAL
```

**Apply the configuration:**
```bash
kubectl apply -f dr.yaml
```

> **Explanation:** `ISTIO_MUTUAL` mode ensures both client and server use mutual TLS with Istio-managed certificates.

## 3. Verify mTLS Configuration

### 3.1 Check PeerAuthentication Policies

```bash
kubectl get peerauthentication -A
```

**Expected output:**
```
NAMESPACE       NAME      MODE     AGE
observability   default   STRICT   30m
```

### 3.2 Check Listener Inbound TLS

Examine the Envoy listener configuration for a specific pod:

```bash
istioctl proxy-config listeners productpage-v1-58d8d7f474-vkwm4 -n observability
```

**Look for indicators of TLS encryption:**
```
0.0.0.0        15006 Trans: tls; App: istio-http/1.0,istio-http/1.1,istio-h2 InboundPassthroughCluster
0.0.0.0        15006 Trans: tls                                              InboundPassthroughCluster
0.0.0.0        15006 Trans: tls; Addr: *:9080                                Cluster: inbound|9080||
```

> ✅ **This confirms inbound traffic is encrypted with TLS**

### 3.3 Describe Pod Configuration

```bash
istioctl x describe pod productpage-v1-58d8d7f474-vkwm4 -n observability
```

**Expected output:**
```
Pod: productpage-v1-58d8d7f474-vkwm4
   Pod Revision: default
   Pod Ports: 9080 (productpage)
--------------------
Service: productpage
   Port: http 9080/HTTP targets pod port 9080
DestinationRule: observability-mtls for "*.observability.svc.cluster.local"
   Traffic Policy TLS Mode: ISTIO_MUTUAL
--------------------
Effective PeerAuthentication:
   Workload mTLS mode: STRICT
Applied PeerAuthentication:
   default.observability
```

> ✅ **This confirms the policy is applied to the pod**

## 4. Verify mTLS with Envoy Logs

### 4.1 Check Envoy Logs for TLS Indicators

```bash
kubectl logs -n observability productpage-v1-58d8d7f474-vkwm4 -c istio-proxy
```

**Look for:**
- `TLS handshake` - Indicates TLS negotiation
- `authenticated principal` - Shows mTLS authentication
- `ISTIO_MUTUAL` - Confirms mutual TLS mode

### 4.2 Connect to Pod and Check TLS

```bash
kubectl exec -n observability -it productpage-v1-58d8d7f474-vkwm4 -- /bin/bash
```

Once inside the container, you can:
- Check environment variables
- View certificate mounts
- Test connectivity with `curl` commands

## 5. Additional Verification Commands

### Check mTLS Statistics

```bash
# View TLS stats for a pod
istioctl proxy-config stats productpage-v1-58d8d7f474-vkwm4 -n observability | grep -i tls
```

### Verify Certificates

```bash
# Check certificate mounts in the sidecar
kubectl exec -n observability productpage-v1-58d8d7f474-vkwm4 -c istio-proxy -- ls -la /etc/certs/
```

### View mTLS Policy Status

```bash
# Check if mTLS is enforced
kubectl exec -n observability productpage-v1-58d8d7f474-vkwm4 -c istio-proxy -- pilot-agent request GET config_dump | jq '.configs[] | select(.@type | contains("PeerAuthentication"))'
```

## 6. mTLS Migration Path

### Phase 1: DISABLE
```yaml
spec:
  mtls:
    mode: DISABLE
```
- No encryption
- Suitable for initial development

### Phase 2: PERMISSIVE
```yaml
spec:
  mtls:
    mode: PERMISSIVE
```
- Accepts both plaintext and mTLS
- Allows gradual migration
- Monitor for plaintext traffic

### Phase 3: STRICT
```yaml
spec:
  mtls:
    mode: STRICT
```
- Only mTLS allowed
- Production recommended
- Maximum security

## 7. Troubleshooting Common Issues

### Issue: Pods Not Starting
```bash
# Check sidecar injection
kubectl get pod <pod-name> -n observability -o yaml | grep -A 5 -i istio

# Check istiod logs
kubectl logs -n istio-system deployment/istiod
```

### Issue: TLS Handshake Failures
```bash
# Check certificate expiry
kubectl exec -n observability <pod-name> -c istio-proxy -- openssl x509 -in /etc/certs/cert-chain.pem -text -noout

# View Envoy logs for TLS errors
kubectl logs -n observability <pod-name> -c istio-proxy | grep -i "tls\|ssl\|handshake"
```

### Issue: PERMISSIVE to STRICT Migration Problems
```bash
# Verify no plaintext traffic
kubectl exec -n observability <pod-name> -c istio-proxy -- pilot-agent request GET stats | grep -i "plaintext"
```

---

## 8. Best Practices

1. **Always start with PERMISSIVE mode** when enabling mTLS on existing workloads
2. **Monitor plaintext traffic** before switching to STRICT mode
3. **Use DestinationRules** to define TLS mode on the client side
4. **Verify certificates** are properly mounted and rotated
5. **Test thoroughly** in staging environment before production deployment
6. **Use service entries** for external services requiring mTLS
