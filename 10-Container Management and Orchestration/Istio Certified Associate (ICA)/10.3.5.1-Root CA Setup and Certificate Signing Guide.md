## Overview

Setting up a self-signed Root Certificate Authority (CA) server on Ubuntu using OpenSSL allows you to sign certificates for:
- Internal applications
- Local websites
- Development servers
- Secure internal communications

---

## 1. Install OpenSSL

Ensure your system package list is updated and OpenSSL utility is installed.

```bash
sudo apt update
sudo apt install openssl ca-certificates -y
```

---

## 2. Establish a Secure Directory Structure

Create a dedicated folder to keep your Root CA files isolated and secure.

```bash
mkdir -p ~/my-root-ca/private
chmod 700 ~/my-root-ca/private
cd ~/my-root-ca
```

---

## 3. Generate the Root CA Private Key

Generate an encrypted RSA private key. You will be prompted to enter a passphrase.

```bash
openssl genrsa -aes256 -out private/ca.key.pem 4096
```

**Passphrase:** `rootca`

> **Note:** A 4096-bit key length ensures robust, long-term cryptographic security. Secure this passphrase carefully; anyone who obtains it can issue certificates on your network.

---

## 4. Create the Self-Signed Root Certificate

Use your private key to generate a self-signed root certificate (`ca.crt.pem`). This command sets validity to 3650 days (approximately 10 years).

```bash
openssl req -new -x509 \
  -key private/ca.key.pem \
  -sha256 -days 3650 \
  -out ca.crt.pem \
  -subj "/C=BD/ST=Dhaka/L=Dhaka/O=DevOpsLab/CN=DevOpsLab Root CA" \
  -addext "basicConstraints=critical,CA:TRUE" \
  -addext "keyUsage=critical,keyCertSign,cRLSign"
```

---

## 5. Verify the Certificate Structure

Confirm your Root CA certificate was generated accurately and contains the correct attributes.

```bash
openssl x509 -noout -text -in ca.crt.pem
```

**Look for:**
- `CA:TRUE` under the X509v3 Basic Constraints section
- This verifies it is successfully registered as a Certificate Authority

---

## 6. Trust the Root CA on Ubuntu Clients

For Ubuntu client machines on your network to trust certificates issued by this CA, install the root certificate into their local trust store.

### Step 6.1: Copy Certificate to Trusted Directory

Copy the certificate to the trusted directory, renaming the extension to `.crt`:

```bash
sudo cp ca.crt.pem /usr/local/share/ca-certificates/my-root-ca.crt
```

### Step 6.2: Update System Trust Store

```bash
sudo update-ca-certificates
```

**Expected output:**
```
Updating certificates in /etc/ssl/certs...
rehash: warning: skipping ca-certificates.crt, it does not contain exactly one certificate or CRL
1 added, 0 removed; done.
Running hooks in /etc/ca-certificates/update.d...
done.
```

### Step 6.3: Verify Installation

```bash
openssl verify -CApath /etc/ssl/certs ca.crt.pem
```

**Expected output:**
```
ca.crt.pem: OK
```

## 7. Create Intermediate CA (Optional)

Creating an Intermediate CA adds an extra layer of security by allowing you to keep the Root CA offline.

### Step 7.1: Generate Intermediate Key

```bash
openssl genrsa -out intermediate.key.pem 4096
```

### Step 7.2: Create CSR for Intermediate CA

```bash
openssl req -new \
  -key intermediate.key.pem \
  -out intermediate.csr.pem \
  -subj "/C=BD/ST=Dhaka/L=Dhaka/O=DevOpsLab/CN=DevOpsLab Intermediate CA"
```

### Step 7.3: Create Extension File

**File:** `intermediate_extension.ext`

```
basicConstraints=critical,CA:TRUE,pathlen:0
keyUsage=critical,keyCertSign,cRLSign
subjectKeyIdentifier=hash
authorityKeyIdentifier=keyid,issuer
```

### Step 7.4: Sign Intermediate Certificate with Root CA

```bash
openssl x509 -req \
  -in intermediate.csr.pem \
  -CA ca.crt.pem \
  -CAkey ca.key.pem \
  -CAcreateserial \
  -out intermediate.crt.pem \
  -days 1825 \
  -sha256 \
  -extfile intermediate_extension.ext
```

### Step 7.5: Verify Intermediate Certificate

```bash
openssl x509 -in intermediate.crt.pem -text -noout
```

**Expected output should include:**
```
X509v3 extensions:
  X509v3 Basic Constraints: critical
    CA:TRUE, pathlen:0
  X509v3 Key Usage: critical
    Certificate Sign, CRL Sign
```

## 8. Generate CSR on Client Machine

### Part 1: Generate Private Key & CSR on Client Machine

On the client machine, generate a private key and a CSR. You will be prompted to enter organization details.

```bash
openssl req -newkey rsa:2048 -nodes -keyout client.key -out client.csr
```

> **Important:** The Common Name must exactly match the domain name or IP address you are securing.

**Example Interactive Input:**
```
Country Name (2 letter code) [AU]:BD
State or Province Name (full name) [Some-State]:Dhaka
Locality Name (eg, city) []:Dhaka
Organization Name (eg, company) [Internet Widgits Pty Ltd]:DevOpsLab
Organizational Unit Name (eg, section) []:ICTD
Common Name (e.g. server FQDN or YOUR name) []:DevOpsLab.anis.com
Email Address []:devops@anis.com

Please enter the following 'extra' attributes
to be sent with your certificate request
A challenge password []:
An optional company name []:
```

---

## 9. Sign CSR Using Root CA

### Part 2: Sign the CSR using your Root CA

Move the generated `client.csr` file securely to the machine hosting your Root CA. Then sign it using your Root CA's certificate and key.

### Step 9.1: Sign the CSR

```bash
openssl x509 -req \
  -in client.csr \
  -CA ca.crt.pem \
  -CAkey ca.key.pem \
  -CAcreateserial \
  -out client.crt \
  -days 365 \
  -sha256
```

### Step 9.2: Transfer the Certificate

Send the newly generated `client.crt` back to the client machine securely.

---

## 10. Install and Verify Client Certificate

### Part 3: Install and Verify

The client machine must trust the certificate you just issued.

### Step 10.1: Install and Bind Certificate

Install and bind the `client.crt` and `client.key` to the client's application or server (e.g., Nginx, Apache, or IIS).

### Step 10.2: Install Root CA on Client/Browser Machines

Install your `rootCA.crt` into the Trusted Root Certification Authorities store on the client/browser machines to prevent security warnings.

## Security Best Practices

1. **Store Root CA securely**: Keep the Root CA private key offline when not in use
2. **Use strong passphrases**: Protect the Root CA key with a strong passphrase
3. **Implement access control**: Restrict access to the Root CA machine
4. **Use Intermediate CA**: Keep Root CA offline and use Intermediate CA for daily signing
5. **Regular backups**: Backup the Root CA directory securely
6. **Monitor certificate usage**: Track issued certificates and revoke when necessary
7. **Use appropriate validity periods**: Balance between security and operational convenience
8. **Document CA procedures**: Maintain detailed records of certificate issuance

## Troubleshooting

### Issue: Certificate Not Trusted
```bash
# Verify certificate trust
openssl verify -CAfile ca.crt.pem client.crt

# Check certificate details
openssl x509 -noout -text -in client.crt
```

### Issue: CSR Rejection
```bash
# Check CSR details
openssl req -noout -text -in client.csr

# Verify private key matches CSR
openssl req -noout -modulus -in client.csr | openssl md5
openssl rsa -noout -modulus -in client.key | openssl md5
```

### Issue: Permission Denied
```bash
# Check directory permissions
ls -la ~/my-root-ca/

# Fix permissions
chmod 600 ~/my-root-ca/private/ca.key.pem
chmod 644 ~/my-root-ca/ca.crt.pem
```
