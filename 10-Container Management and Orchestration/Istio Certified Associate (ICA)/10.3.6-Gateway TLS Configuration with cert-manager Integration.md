## 1. Overview

This guide explains how to implement Gateway TLS configuration with certificates using cert-manager integration. The certificates are automatically created and managed by cert-manager, eliminating manual certificate management.

---

## 2. Architecture Flow

```
Certificate YAML Applied
        ↓
cert-manager issues certificate
        ↓
Secret "lab-tls" is created automatically
        ↓
Gateway references lab-tls Secret
        ↓
TLS termination happens at Gateway
        ↓
HTTPRoute sends traffic to backend service
```

### Full Flow Diagram

```
Internet (Client)
   ↓
HTTPS Request (devsecopslab.com)
   ↓
Gateway (TLS Termination)
   ↓
HTTPRoute (Routing Rules)
   ↓
Backend Service (nginx.bookinfo)
```

## 3. Prerequisites

- cert-manager installed and configured
- ClusterIssuer (root-ca-issuer) created and ready
- Istio installed with Gateway API support

## 4. Create Certificate with cert-manager

### Certificate Resource Definition

**File:** `cert-manager.yaml`

```yaml
apiVersion: cert-manager.io/v1
kind: Certificate
metadata:
  name: devsecopslab-cert
  namespace: gateway-system
spec:
  secretName: lab-tls
  dnsNames:
  - devsecopslab.com
  issuerRef:
    name: root-ca-issuer
    kind: ClusterIssuer
```

### Apply Certificate

```bash
kubectl apply -f cert-manager.yaml
```

### What Happens:
1. cert-manager detects the Certificate resource
2. Issues a certificate for `devsecopslab.com`
3. Automatically creates the `lab-tls` Secret
4. Secret contains `tls.crt` and `tls.key`

> **Note:** The Secret is NOT created manually. It is automatically created from the Certificate resource by cert-manager.

## 5. Create Gateway with HTTPS Listener

### Gateway Resource Definition

**File:** `gateway-https.yaml`

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: nginx-gateway
  namespace: gateway-system
spec:
  gatewayClassName: istio
  listeners:
  - name: http
    protocol: HTTP
    port: 80
    allowedRoutes:
      namespaces:
        from: All
  - name: tls-test-https
    protocol: HTTPS
    port: 443
    hostname: devsecopslab.com
    allowedRoutes:
      namespaces:
        from: All
    tls:
      mode: Terminate
      certificateRefs:
      - group: ""
        kind: Secret
        name: lab-tls
```

### Apply Gateway

```bash
kubectl apply -f gateway-https.yaml
```

### Explanation of Listener:

| Field | Description | Usage |
|-------|-------------|-------|
| `name: tls-test-https` | Logical identifier for the listener | Used internally by Gateway controller and HTTPRoute binding |
| `protocol: HTTPS` | Protocol type for the listener | Defines TLS termination |
| `port: 443` | Standard HTTPS port | Entry point for encrypted traffic |
| `hostname: devsecopslab.com` | Domain for this listener | Matches incoming requests |
| `tls.mode: Terminate` | TLS termination mode | Gateway terminates TLS and forwards plain HTTP |

## 6. Create HTTPRoute for TLS Traffic

### HTTPRoute Resource Definition

**File:** `httproute-https.yaml`

```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: nginx-route
  namespace: bookinfo
spec:
  parentRefs:
  - name: nginx-gateway
    namespace: gateway-system
  hostnames:
  - devsecopslab.com
  rules:
  - matches:
    - path:
        type: PathPrefix
        value: /
    backendRefs:
    - name: nginx
      port: 80
```

### Apply HTTPRoute

```bash
kubectl apply -f httproute-https.yaml
```

## 10. Trust Root CA for Development

### Option 1: Trust via Root CA Server

If you have the Root CA certificate from the server:

```bash
cp rootCA.crt /usr/local/share/ca-certificates/devsecopslab.crt
update-ca-certificates
```

### Option 2: Extract Root CA from Secret

Extract root CA from the cert-manager secret:

```bash
kubectl get secret inter-ca-secret -n cert-manager -o jsonpath="{.data.tls\.crt}" | base64 -d > root-ca.crt
cp root-ca.crt /usr/local/share/ca-certificates/devsecopslab.crt
update-ca-certificates
```

### Option 3: Verify Root CA Trust

```bash
openssl verify rootCA.crt
```

**Expected output:**
```
rootCA.crt: OK
```

## 11. Best Practices

### Production Order

1. **Install cert-manager**
   - Verify pods are running
   - Check CRDs are installed

2. **Verify ClusterIssuer**
   - Check status is `READY`
   - Confirm CA secret exists

3. **Create ClusterIssuer**
   - Set correct secret name
   - Verify issuer is ready

4. **Apply Certificate YAML**
   - Ensure DNS names are correct
   - Verify certificate is issued

5. **Attach Secret to Gateway**
   - Reference the automatically created secret
   - Configure TLS termination mode

### Production Checklist

- [ ] cert-manager installed and running
- [ ] ClusterIssuer created and ready
- [ ] Certificate created and issued
- [ ] Secret automatically created
- [ ] Gateway references correct secret
- [ ] TLS mode set to `Terminate`
- [ ] Hostname configured correctly
- [ ] HTTPRoute bound to Gateway
- [ ] TLS handshake successful
- [ ] Trust chain verified

## 13. Troubleshooting

### Issue: Certificate Not Ready

```bash
# Check certificate status
kubectl describe certificate devsecopslab-cert -n gateway-system

# Check cert-manager logs
kubectl logs -n cert-manager deployment/cert-manager
```

### Issue: Secret Not Created

```bash
# Check CertificateRequest
kubectl get certificaterequest -n gateway-system

# Check if issuer is ready
kubectl get clusterissuer root-ca-issuer -o yaml
```

### Issue: Gateway Not Programmed

```bash
# Check gateway status
kubectl describe gateway nginx-gateway -n gateway-system

# Verify secret exists in gateway namespace
kubectl get secret lab-tls -n gateway-system
```

### Issue: TLS Handshake Fails

```bash
# Check TLS configuration
curl -v https://devsecopslab.com

# Verify certificate details
openssl s_client -connect devsecopslab.com:443 -servername devsecopslab.com

# Check Gateway logs
kubectl logs -n istio-system deployment/istio-ingressgateway
```

### Issue: Self-Signed Certificate Not Trusted

```bash
# Add root CA to trust store
cp rootCA.crt /usr/local/share/ca-certificates/
update-ca-certificates

# Verify trust
openssl verify -CApath /etc/ssl/certs rootCA.crt
```

## 14. Understanding listener.name

### Key Points

| Aspect | Description |
|--------|-------------|
| **NOT used by core Kubernetes objects** | listener.name is not a runtime service object |
| **IS used by Gateway controller** | Internal tracking and management |
| **IS used by HTTPRoute binding** | Can be referenced via `sectionName` |
| **IS used internally** | Routing logic and policies |
| **Acts as logical identifier** | Not exposed to external users |

### Example Usage

```yaml
httpRoute:
  parentRefs:
  - name: nginx-gateway
    namespace: gateway-system
    sectionName: tls-test-https  # References the listener name
```
