 
# Loki Installation and Configuration Guide

## Complete Setup with Fluent Bit Log Shipping

---

## Part 1: Loki Installation

### Install Loki Using Sample Application YAML

```bash
kubectl apply -f samples/addons/loki.yaml
```

### Install a StorageClass

You must install a storage backend for Loki to persist logs.

#### For Bare-Metal / VM Clusters:

**Install local-path provisioner:**
```bash
kubectl apply -f https://raw.githubusercontent.com/rancher/local-path-provisioner/master/deploy/local-path-storage.yaml
```

**Set as default StorageClass:**
```bash
kubectl patch storageclass local-path -p '{"metadata":{"annotations":{"storageclass.kubernetes.io/is-default-class":"true"}}}'
```

## Part 2: Troubleshooting PVC Issues

If Loki pod is stuck, follow these steps:

### Step 1: Delete Stuck PVC
```bash
kubectl delete pvc storage-loki-0 -n istio-system
```

### Step 2: Delete Loki Pod (Forces Re-creation)
```bash
kubectl delete pod loki-0 -n istio-system
```

> **Note:** Because it's a StatefulSet, it will recreate automatically.

### Step 3: Watch Provisioning
```bash
kubectl get pvc -n istio-system -w
```

## Part 5: Fluent Bit Installation (Log Shipper)

### 5.1 Service Account + RBAC

Create a file named `fluent-bit-rbac.yaml`:

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: fluent-bit
  namespace: logging
---
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: fluent-bit
rules:
  - apiGroups: [""]
    resources:
      - pods
      - namespaces
    verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: fluent-bit
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: fluent-bit
subjects:
  - kind: ServiceAccount
    name: fluent-bit
    namespace: logging
```

**Apply the RBAC configuration:**
```bash
kubectl apply -f fluent-bit-rbac.yaml
```

---

### 5.2 ConfigMap (Fluent Bit Configuration)

Create a file named `fluent-bit-configmap.yaml`:

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: fluent-bit-config
  namespace: logging
data:
  fluent-bit.conf: |
    [SERVICE]
        Flush        5
        Daemon       Off
        Log_Level    info
        Parsers_File /fluent-bit/etc/parsers.conf
     
    [INPUT]
        Name tail
        Path /var/log/containers/*.log
        Parser cri
        Tag kube.*

    [FILTER]
        Name kubernetes
        Match kube.*
        
        # API server endpoint
        kube_url            https://kubernetes.default.svc:443
        kube_ca_file        /var/run/secrets/kubernetes.io/serviceaccount/ca.crt
        kube_token_file     /var/run/secrets/kubernetes.io/serviceaccount/token

        # Metadata enrichment options
        merge_log           on
        merge_log_trim      on
        keep_log            off

        # Include additional metadata
        k8s-logging.parser  on
        k8s-logging.exclude off

        # Labels and annotations to include
        labels              on
        annotations         off

    # Add environment metadata
    [FILTER]
        name         record_modifier
        match        kube.*
        record       cluster_name ${CLUSTER_NAME}
        record       environment production
    
    [OUTPUT]
        Name loki
        Match *
        Host loki.istio-system.svc.cluster.local
        Port 3100
        Labels job=fluentbit,namespace=$kubernetes['namespace_name'],pod=$kubernetes['pod_name'],service=$kubernetes['labels']['app']
        Line_Format json
```

**Apply the ConfigMap:**
```bash
kubectl apply -f fluent-bit-configmap.yaml
```

---

### 5.3 DaemonSet

Create a file named `fluent-bit-daemonset.yaml`:

```yaml
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: fluent-bit
  namespace: logging
spec:
  selector:
    matchLabels:
      app: fluent-bit
  template:
    metadata:
      labels:
        app: fluent-bit
    spec:
      serviceAccountName: fluent-bit
      containers:
        - name: fluent-bit
          image: cr.fluentbit.io/fluent/fluent-bit:2.2
          volumeMounts:
            - name: varlog
              mountPath: /var/log
            - name: config
              mountPath: /fluent-bit/etc/
      volumes:
        - name: varlog
          hostPath:
            path: /var/log
        - name: config
          configMap:
            name: fluent-bit-config
```

**Apply the DaemonSet:**
```bash
kubectl apply -f fluent-bit-daemonset.yaml
```

## Part 6: Understanding the Fluent Bit Pipeline

Before diving into filters, you need to understand where they fit in the processing pipeline.

```
┌──────────────┐
│   INPUT       │
│ (Log Source)  │
└──────┬───────┘
       │
       ▼
┌──────────────┐
│   PARSER      │
│ (Structure   │
│  raw logs)    │
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ FILTER 1      │
│ (Kubernetes   │
│ metadata add) │
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ FILTER 2      │
│ (Grep/Drop/   │
│ Modify logs)  │
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ FILTER N      │
│ (Extra rules) │
└──────┬───────┘
       │
       ▼
┌──────────────┐
│   OUTPUT      │
│ (Loki/ES/etc) │
└──────────────┘
```

## Part 7: Fluent Bit Configuration Parameters Explained

### SERVICE Section Parameters

| Parameter | Value | Description |
|-----------|-------|-------------|
| `Flush` | `5` | Sends logs to output every 5 seconds (Lower = faster delivery, Higher = better performance) |
| `Daemon` | `Off` | Runs in foreground for Kubernetes (Off = K8s mode, On = background process) |
| `Log_Level` | `info` | Controls internal logs (error, warn, info, debug) |
| `Parsers_File` | `/fluent-bit/etc/parsers.conf` | Loads parsing rules (CRI, JSON, etc.) |

### Log Level Meanings

| Level | Meaning |
|-------|---------|
| `error` | Only errors |
| `warn` | Warnings |
| `info` | Normal logs |
| `debug` | Deep troubleshooting |

### Monitoring & Control API Parameters

| Parameter | Description |
|-----------|-------------|
| `http_server` | Switches on integrated internal web server for health metrics |
| `http_listen` | Binds telemetry server API to specific local interface address |
| `http_port` | Designates precise TCP port allocation for server endpoint |
| `hot_reload` | Enables configuration updates in real-time using SIGHUP signals |

## Part 8: Pipeline Summary

| Section | Purpose |
|---------|---------|
| `[SERVICE]` | Fluent Bit engine settings |
| `[INPUT]` | Log collection |
| `[FILTER]` | Processing/enrichment |
| `[OUTPUT]` | Sending logs |

### How Log Records Flow

Each log record flows through the pipeline sequentially. Filters can:

- Modify existing fields
- Add new fields
- Remove unwanted fields
- Drop entire records
- Route records to different outputs

## Part 10: Common Troubleshooting

### Fluent Bit Cannot Connect to Loki

**Verify Loki service is running:**
```bash
kubectl get svc -n istio-system loki
```

**Test connectivity from Fluent Bit pod:**
```bash
kubectl exec -n logging <fluent-bit-pod> -- wget -O- http://loki.istio-system.svc.cluster.local:3100/ready
```

### No Logs Appearing in Grafana

**Check Fluent Bit output logs:**
```bash
kubectl logs -n logging <fluent-bit-pod> | grep -i error
```

**Verify Loki is receiving logs:**
```bash
kubectl logs -n istio-system <loki-pod> | grep -i "received"
```

### PVC Issues

**Check PVC status:**
```bash
kubectl get pvc -n istio-system
```

**Describe PVC for details:**
```bash
kubectl describe pvc storage-loki-0 -n istio-system
```

## Best Practices

1. **Resource Limits**: Set appropriate CPU/memory limits for Fluent Bit DaemonSet
2. **Log Rotation**: Ensure host system has proper log rotation configured
3. **Sampling**: Use sampling filters for high-volume environments
4. **Monitoring**: Monitor Fluent Bit metrics for backpressure
5. **Retention**: Configure Loki retention policies to manage storage costs
