# A Comprehensive Guide/Discussion on HTTPS TLS SSL

## Introduction to SSL

SSL (Secure Sockets Layer) works almost like a passport, serving two purposes:
1. Permits access to encrypted information through Public Key Infrastructure
2. Helps authenticate the identity of the owner of the SSL certificate

SSL, also known as TLS (Transport Layer Security), is widely used on websites regardless of type — whether it's a simple blog or a shopping portal.

### HTTP vs HTTPS

If you've noticed URLs of websites, they either show HTTPS or HTTP. The difference between the two is the 'S' at the end. **HTTPS** indicates that the website is secured by an SSL certificate that helps secure communication between your web browser and the website you're visiting.

In other words, SSL is a security technology that offers a secure connection between the web server and web browser. It assures that the data transferred is secured and original.

### SSL/TLS Versions Timeline

```
SSL 1.0 → SSL 2.0 → SSL 3.0 → TLS 1.0 → TLS 1.1 → TLS 1.2 → TLS 1.3
(1995)   (1996)   (1999)   (2006)   (2008)   (2018)
```

> **Note:** TLS certificates are still known as SSL certificates. When someone purchases an SSL Certificate, they are actually purchasing the latest TLS certificates with options like RSA or DSA encryption.

## Why SSL?

In today's digital world where cyber-attacks are more prevalent, securing a website with an SSL/TLS Certificate has become a norm. SSL/TLS Certificates help achieve two main purposes of security:

- **Encryption**
- **Authentication**

### Other Reasons for SSL

- Secures communication between website and customer's web-browser
- Secures communication on corporate intranet
- Secures email communications
- Secures transfer of information between internal and external servers
- Secures information sent and received between mobile devices

### The Data Transmission Problem

When you enter information, it goes through multiple devices before reaching its destination. Using the `tracert` command:

**For Windows Users:**
1. Go to **Start** and Click on **Run**
2. Type "**cmd**" and hit Enter
3. Type `tracert domain-name.com`
4. Hit **Enter**

You'll see a list of devices (nodes) where your data gets recorded — typically 20-30 listings. This includes sensitive information like credit card numbers, passwords, and financial details.

**This is where SSL/TLS comes in** — it encrypts such data using a complex key exchange system, providing secure transaction between web browser and server.

## Features & Benefits of SSL Certificate Security

### SSL Protects Data Through Encryption

SSL/TLS Certificate protects website visitors' sensitive information (login details, passwords, account details, credit card numbers) by establishing a secured encrypted tunnel between Client (Web Browser) and Server. SSL/TLS Certificates come equipped with strong **256-bit encryption standard** that cannot be easily cracked.

### SSL Confirms Your Identity

SSL certificates help verify website authenticity. SSL providers verify the legitimacy of the person or organization purchasing the SSL certificate through processes set by CA/B (a joint forum of CAs and Web browsers).

### SSL Offers Non-Repudiation

Combination of Encryption, Integrity, and Authentication establishes non-repudiation — meaning none of the parties in a secured transaction can legally claim their communication came from someone else.

## What Could Go Wrong Without SSL/TLS?

If an SSL/TLS Certificate is not installed, attacks like these can happen easily:

| Attack Type | Description |
|-------------|-------------|
| **APT (Advanced Persistent Threats)** | Advanced attacks by proficient, well-funded external entities using techniques like drive-by-downloads and SQL injection |
| **Man-in-the-Middle (MITM)** | Attacker secretly changes data between two parties who think they're directly communicating |
| **Protocol Attacks** | Exploits weaknesses in communication protocols |

## How Does SSL Work?

### Basic Process (Layman's Perspective)

1. **Browser/Server connects** to an SSL secured website (web server) and requests to identify the web server
2. **Web server sends** a copy of the SSL/TLS Certificate to the web browser/server
3. **Browser/Server checks** whether the SSL certificate is trustworthy
   - If trustworthy, it sends a message to the web server
4. **Web server sends back** a digitally signed acknowledgment to start an SSL encrypted session
5. **Encrypted session begins** and encrypted data is shared

## SSL Architecture and Protocols

SSL/TLS operates directly on top of TCP. Higher-level protocols (like HTTP) remain unchanged while providing a secured connection — HTTP is identical to HTTPS beneath the SSL layer.

### SSL Protocol Stack

```
┌─────────────────────────────────────────────┐
│         Application Layer (HTTP)            │
├─────────────────────────────────────────────┤
│         SSL/TLS Protocols                    │
│  ┌─────────────┐ ┌──────────┐ ┌───────────┐ │
│  │  Handshake  │ │  Alert   │ │ Change    │ │
│  │  Protocol   │ │ Protocol │ │ CipherSpec│ │
│  └─────────────┘ └──────────┘ └───────────┘ │
├─────────────────────────────────────────────┤
│           SSL Record Protocol               │
├─────────────────────────────────────────────┤
│                 TCP                         │
└─────────────────────────────────────────────┘
```

### SSL Handshake Protocol

SSL Handshake Protocol establishes an HTTPS connection — all important work is done here. It establishes a secure channel between server and client.

**Main purposes:**
- Exchange of Encryption Capabilities
- Authentication of the SSL/TLS Certificate
- Exchanging or Generating a Session Key

#### SSL/TLS Handshake Steps (TLS 1.2)

| Step | Message | Action |
|------|---------|--------|
| 1 | **ClientHello** | Client initiates handshake, recommending SSL parameters |
| 2 | **ServerHello** | Server responds with selected SSL parameters; terminates if no common parameters |
| 3 | **Certificate** | Server sends SSL Certificate chain; Client verifies digital signature, certificate chain, expiry, domain name |
| 4 | ServerKeyExchange | Optional message for additional key exchange data |
| 5 | ServerHelloDone | Server concludes SSL negotiation messages |
| 6 | ClientKeyExchange | Client sends session key info encrypted with server's public key |
| 7 | ChangeCipherSpec | Client instructs server to activate negotiated SSL parameters |
| 8 | Finished | Client asks server to verify successful SSL negotiations |
| 9 | ChangeCipherSpec | Server instructs client to activate negotiated SSL parameters |
| 10 | Finished | Server instructs client to verify successful SSL negotiations |

### SSL Record Protocol

The SSL Record Protocol handles encryption of all messages, offering a common format for Handshake, Alerts, Change Cipher Spec, and Application Protocols.

**Structure:**
- Summarized data
- Message type
- Version
- Length
- Digital signature (8 bytes total)

### SSL Alert Protocol

Handles questionable packets with three alert message types:
- **Warning** - Session restricted
- **Critical** - Session restricted
- **Fatal** - Session terminated

### Change Cipher Spec Protocol

Signals transitions in cipher strategies. Contains a single message carrying a single value byte of 1, notifying the receiver that successive records will stay protected under newly exchanged CipherSpec and keys.

## Authentication

Authentication is identity confirmation — one of the processes used for network interactions involving recognition of one party by another.

- **Server Authentication** - Identification of a server by a client
- **Client Authentication** - Identification of a client by a server

### How Client Authentication by Server Works

1. Client software manages a database with private keys corresponding to public keys in issued certificates
2. Client unlocks database, retrieves private key, uses it to sign randomly generated data
3. Randomly generated data and user's certificate are sent across the network
4. Server authenticates user identity using signed data and certificate
5. Server may perform additional authentication (LDAP directory check, authorization evaluation)

## Encryption

Encryption makes data incomprehensible to ensure confidentiality. Modern encryption technologies like SSL/TLS are widely used over the internet.

### Types of Encryption

#### Public-Key (Asymmetric) Encryption

Uses two different keys:
- **Public Key** - Accessible to everyone
- **Private Key** - Remains private with the key owner

| Feature | Description |
|---------|-------------|
| Keys | Mathematically related but not identical |
| Public Key Use | Encrypts data |
| Private Key Use | Decrypts data |
| Algorithms | RSA, DSA |
| Security | Almost impossible to derive private key from public key |

#### Bulk/Symmetric Encryption

Uses one secret key to cipher and decipher information.

| Feature | Description |
|---------|-------------|
| Key | Single secret key (numbers, words, or random alphabets) |
| Requirement | Recipient must know the secret key |
| Algorithms | Blowfish, AES, DES, RC4, RC5, RC6 |
| Widely Used | AES-128, AES-192, AES-256 |

### Asymmetric vs Symmetric Encryption

| Aspect | Symmetric | Asymmetric |
|--------|-----------|------------|
| Key | Single key shared among recipients | Pair of public-private keys |
| Age | Older method | Newer method |
| Key Sharing | Required | Eliminated |
| Speed | Faster | Slower |

### How SSL/TLS Uses Both

1. Copy of asymmetric public key is sent to the Server
2. Browser creates symmetric session key, encrypts it with asymmetric public key, sends to server
3. Server uses asymmetric private key to decrypt the encrypted session key
4. Server & Browser use symmetric session key to encrypt/decrypt all transmitted data
5. Session key is used only for that session (new key for each session)

## Cipher Suites

A cipher suite requires different ingredients — protocols and algorithms instead of food items.

### Components of Cipher Suites

| Component | Description | Examples |
|-----------|-------------|----------|
| **Key Exchange Algorithm** | Secures exchange of symmetric keys over networks | ECDHE, ECDH, DHE, RSA |
| **Authentication Algorithm** | Verifies user identity | ECDSA, RSA, DSA |
| **Bulk Encryption Algorithm** | Ensures secure data transmission | CAMELLA, AES, 3DES |
| **MAC Algorithm** | Verifies and authenticates received messages | MD5, SHA |

> **Note:** GCM (Galois Counter Mode) is a block cipher mode offering authenticated encryption/decryption, widely used for performance and efficiency (e.g., AES-GCM with 256-bit keys).

### Example Cipher Suite

```
ECDHE-ECDSA-AES256-GCM-SHA384
│      │      │        │
│      │      │        └── MAC Algorithm (SHA384)
│      │      └─────────── Bulk Encryption (AES256-GCM)
│      └────────────────── Authentication (ECDSA)
└───────────────────────── Key Exchange (ECDHE)
```

### How Cipher Suites Work

During Handshake, server and client exchange a list of prioritized cipher suites and decide to use the one best supported by both. The cipher suite affects:
- Security
- Speed
- Compatibility

## Public Key Infrastructure (PKI)

PKI (Public Key Infrastructure) is a set of rules, procedures, and policies needed to create, distribute, manage, store, use, or revoke certificates and manage public-key encryption.

### Key Components of PKI

| Component | Description |
|-----------|-------------|
| **Certificate Policy** | Security requirements defining hierarchy and structure of PKI environment |
| **Root Certificate Authority (CA)** | "Main root of trust" accountable for identity authentication |
| **Subordinate/Intermediate CA** | Certified through root CA; signs and issues digital certificates |
| **Certificate Database** | Stores records of certificates |
| **Revocation Services** | Post updated CRLs or OCSP responders for revocation lookup |
| **Digital Certificates** | Digital identity offering security and authentication |

### PKI Hierarchy

```
                    ┌─────────────┐
                    │   Root CA   │ (Offline - High Security)
                    └──────┬──────┘
                           │
                    ┌──────▼──────┐
                    │Intermediate │
                    │     CA      │
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
         ┌────▼────┐  ┌─────▼─────┐  ┌───▼────┐
         │ Device  │  │  Device   │  │ Device │
         │ Cert    │  │  Cert     │  │ Cert   │
         └─────────┘  └───────────┘  └────────┘
```

**Benefits of Hierarchical PKI:**
- Allows revocation or denial of access at selected levels if private key compromise occurs
- Single sub-CA can generate millions of device certificates
- Root CA can remain offline for increased security

## TLS 1.3: Faster and More Secure

TLS 1.3 (RFC 8446) is the seventh iteration of the SSL/TLS protocol, taking ~28 drafts and ten years to be defined after TLS 1.2.

### TLS 1.3 Timeline

```
2014 ──→ 2015 ──→ 2016 ──→ 2017 ──→ 2018 (Release)
Draft 1                     Final (Draft 28)
```

### TLS 1.3 Handshake Steps

| Step | Message | Action |
|------|---------|--------|
| 1 | ClientHello + Supported Cipher Suites + Key Share | Client initiates with guessed key agreement protocol and key share |
| 2 | ServerHello + Key Share + Certificate + ServerFinished | Server replies with chosen protocol, key share, certificate, and finished message |
| 3 | Generate Keys + ClientFinished | Client verifies certificate, generates keys, sends finished message |

### Benefits of TLS 1.3 over TLS 1.2

| Benefit | Description |
|---------|-------------|
| **Security** | Removed problematic options and outdated ciphers; no renegotiation attacks |
| **Privacy** | Perfect Forward Secrecy (PFS) enabled by default |
| **Performance** | Single round-trip handshake (vs. two round-trips in TLS 1.2); 0-RTT for resumed sessions |

### Removed Vulnerable Algorithms and Ciphers

- RSA Key Exchange
- RC4 Stream Cipher
- CBC (Block) Mode Ciphers
- MD5 Algorithm
- SHA-1 Hash Function
- Short-lived Diffie-Hellman groups
- DES, 3-DES
- Export Strength Ciphers

### Browser Support for TLS 1.3

| Browser | Minimum Version |
|---------|-----------------|
| Google Chrome | 67+ |
| Mozilla Firefox | 61+ |
| Apple (Mac OS/iOS) | 10.3 / iOS 11 |

## Roots, Intermediates, and Leaf Certificates

### Root Certificate
- Self-signed public key certificate
- Forms basis of X.509 based PKI
- Longer lifespan than leaf SSL certificates
- One CA can have many root certificates

### Intermediate Certificate
- Provides security to the Root Certificate
- Works as trusted root for issuing leaf certificates
- Creates chain: Root CA → Intermediate → Leaf Certificate

### Leaf Certificate (End-Entity Certificate)
- Last non-CA certificate in the chain
- Contains public key used by users
- Not directly issued by Root CA (security measure)

## Certificate Chain / Chain of Trust

A chain of trust is established by validating components from the end entity to the root certificate.

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│    Root      │────▶│ Intermediate│────▶│    Leaf      │
│ Certificate  │     │ Certificate │     │ Certificate  │
│ (Trust Anchor)│     │              │     │ (End-Entity) │
└──────────────┘     └──────────────┘     └──────────────┘
```

## Types of SSL Certificates

### By Validation Level

| Type | Description | Vetting Level | Best For |
|------|-------------|---------------|----------|
| **DV (Domain Validation)** | Entry level; proves domain ownership | Minimal (minutes) | Small websites, bloggers |
| **OV (Organization Validation)** | Light business vetting; displays verified business details | Moderate | Business websites |
| **EV (Extended Validation)** | Highest authentication; stringent vetting (1-5 days) | Most stringent | Banks, e-commerce |

### By Functionality

| Type | Protection | Use Case |
|------|------------|----------|
| **Single Domain** | One specific domain (www + non-www) | Single website |
| **Multi-Domain/SAN** | Multiple domains | Organizations with multiple domains |
| **Wildcard** | Main domain + unlimited subdomains | One domain with many subdomains |
| **Multi-Domain Wildcard** | Multiple domains + their subdomains | Complete web presence |

**Example - Wildcard SSL:**
- Purchased for: `www.My-New-Domain.com`
- Also protects: `mail.My-New-Domain.com`, `forum.My-New-Domain.com`, `blog.My-New-Domain.com`

> **Note:** EV Wildcard SSL/TLS Certificates are not available.

|---------|-------------------|

## SSL Certificate Lifecycle Management

### Lifecycle Stages

```
Enrollment → Validation → (Revocation) → Renewal → Abolishing
                ↓
           Inspection
```

| Stage | Description |
|-------|-------------|
| **Enrollment** | User requests certificate; CA verifies information and issues certificate |
| **Validation** | Certificate status verified against Certificate Revocation List (CRL) |
| **Revocation** | Certificate invalidated before expiry (compromised key, employee departure, etc.) |
| **Renewal** | New certificate for same public key (within 90 days of expiration) |
| **Abolishing** | Certificate and backups destroyed when no longer needed |
| **Inspection** | Tracking creation, revocation, and expiration of certificates |

### Certificate Signing Request (CSR)

A CSR is a request block in encoded text format provided to a CA when applying for an SSL/TLS Certificate.

**CSR Information Fields:**

| Field | Description | Example |
|-------|-------------|---------|
| Common Name | FQDN of your server | `*.domain.com` or `mail.domain.com` |
| Organization | Legal name of organization | ABC Example, Inc. |
| Organization Unit | Division handling certificate | IT Department |
| City/Locality | City where organization is located | Spokane |
| State/Country/Region | State or region | Washington |
| Country | Two-letter ISO code | US |
| Email Address | Contact email | webmaster@google.com |
| Public Key | Automatically created | — |

**CSR Format:**
```
-----BEGIN CERTIFICATE REQUEST-----
[Base64-encoded certificate information]
-----END CERTIFICATE REQUEST-----
```

## SSL Certificate Expiration

### Why Do SSL Certificates Expire?

1. **Keep websites updated** with latest security standards
2. **Enforce staying updated** with latest TLS ciphers and versions
3. **Prevent outdated encryption standards** that create loopholes

### Consequences of Expired Certificates

- Invalid certificate
- Website no longer considered safe
- Warning/error messages to users
- Loss of trust
- Decline in sales and revenue
- Adverse effect on brand reputation

### Prevention Tips

1. Find a good certificate management platform (Comodo, DigiCert offer enterprise solutions)
2. Set up CAA records to restrict who can issue SSL for your domains
3. Track email addresses used for purchasing SSL/TLS Certificates
4. Set reminders (renew within 90 days of expiration)

> **Important:** Renewing after expiration means purchasing all over again — your website remains unprotected until the process completes.

## SSL Revocation

**Revocation** means invalidating an already issued certificate before its expiration period.

### Reasons for Revocation

- Person handling certificates leaves company or changes role
- Improper certificate issued by CA
- Domain no longer owned by its owner
- Original certificate replaced with different issuer
- Certificate owner stops operations or site is no longer active

### Revocation Checking Methods

| Method | Description | Pros | Cons |
|--------|-------------|------|------|
| **CRL (Certificate Revocation Lists)** | Text file with list of revoked certificates signed by CA | Simple to use | Large file size, manual import needed, infrequent updates |
| **CRLDPs (CRL Distribution Points)** | CRL automatically read from online source (LDAP/HTTP/S) | Automates CRL retrieval | Still has size problem |
| **OCSP (Online Certificate Status Protocol)** | Client fetches status of specific certificate only | Efficient, solves size problem | Privacy leak (logs IPs), slower, additional round trips |

### OCSP Stapling

An alternative approach where the server queries its own certificate status and "staples" the response to the handshake.
