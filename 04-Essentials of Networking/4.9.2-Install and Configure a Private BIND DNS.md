# How to Install and Configure a Private BIND DNS Server on Ubuntu 22.04

DNS (Domain Name System) is a fundamental network service that every Linux admin and DevOps engineer should be familiar with. DNS is the "phone book of the Internet" that resolves IP addresses like `198.51.100.11` to FQDNs (fully qualified domain names) like `example.com`. The BIND (Berkeley Internet Name Domain) DNS suite is the de-facto DNS server implementation for \*nix systems.

## What is BIND DNS?

BIND is a full-featured, scalable, and open-source software suite for DNS services. The current implementation is **BIND 9**. While there was a BIND 10 project, the ISC stopped development on it in 2014. Today, the BIND 10 project became the Bundy DNS server project, which is currently in hibernation.

The majority of the DNS servers on the internet use BIND 9, and it is a staple in \*nix environments. For Linux administrators, BIND 9 is a robust DNS solution for multiple use cases and includes support for TSIG (transaction signature), DNSSEC, and IPv6.

Now let's dive into configuring a private BIND DNS server on Ubuntu 22.04.

## Our Lab Environment

For our example, we'll configure a single DNS server and two clients.

| Role       | Hostname        | IP Address     |
|------------|----------------|----------------|
| DNS Server | `bindserver`    | `172.17.12.77` |
| Client 1   | `client1`       | `172.17.12.60` |
| Client 2   | `client2`       | `172.17.12.54` |
| Domain     | `anis.example`  | Subnet: `172.17.12.0/24` |

## Prerequisites

Before you begin, you'll need:

- Access to the terminal of three Linux systems on the same LAN (local area network) you can use for testing. We'll use Ubuntu 22.04 LTS.
- Root/sudo privileges on all machines.

## Step 1: Install BIND 9 on the DNS Server

First, update your system:

```bash
sudo apt update -y && sudo apt upgrade -y
```

> **Note:** The `&&` ensures that `apt upgrade` only runs if `apt update` is successful.

Next, install the BIND 9 packages:

```bash
sudo apt install bind9 bind9utils bind9-doc -y
```

Verify that BIND is running:

```bash
sudo systemctl status bind9
```

## Step 2: Configure the BIND Options File

Edit the main options file:

```bash
sudo nano /etc/bind/named.conf.options
```

Replace the contents with the following configuration:

```bash
acl "trusted" {
        172.17.12.77;    # ns1 (bindserver)
        172.17.12.54;    # client2
        172.17.12.60;    # client1
};

options {
        directory "/var/cache/bind";

        allow-query { trusted; localhost; };

        // Allow recursion for trusted clients
        recursion yes;
        allow-recursion { trusted; localhost; };

        // Listen on all interfaces
        listen-on { any; };

        // Forwarding DNS (optional but common)
        forwarders {
                8.8.8.8;
                8.8.4.4;
        };

        dnssec-validation auto;
        auth-nxdomain no;
        listen-on-v6 { any; };
};
```

Check the syntax of the file:

```bash
sudo named-checkconf /etc/bind/named.conf.options
```

## Step 3: Configure the Local Zone File

Edit the local zone configuration:

```bash
sudo nano /etc/bind/named.conf.local
```

Add the forward and reverse zones:

```bash
zone "anis.example" {
    type master;
    file "/etc/bind/zones/anis.example";
};

zone "12.17.172.in-addr.arpa" {
    type master;
    file "/etc/bind/zones/anis.example.rev";
};
```

Check the syntax:

```bash
sudo named-checkconf /etc/bind/named.conf.local
```

## Step 4: Create the Zones Directory

```bash
sudo mkdir /etc/bind/zones
```

## Step 5: Create the Forward Zone File

Copy the default zone file as a template:

```bash
sudo cp /etc/bind/db.local /etc/bind/zones/anis.example
```

Edit the forward zone file:

```bash
sudo nano /etc/bind/zones/anis.example
```

Replace the contents with:

```bind
$TTL    604800
@       IN      SOA     ns1.anis.example. root.ns1.anis.example. (
                              2026051701 ; Serial
                              604800     ; Refresh
                               86400     ; Retry
                             2419200     ; Expire
                              604800 )   ; Negative Cache TTL
;

@       IN      NS      ns1.anis.example.

ns1     IN      A       172.17.12.77

anis.example.    IN     MX   10   mail.anis.example.

www     IN      A       172.17.12.60

ftp     IN      CNAME   www.anis.example.
```

Check the forward zone configuration:

```bash
sudo named-checkzone anis.example /etc/bind/zones/anis.example
```

## Step 6: Create the Reverse Zone File

Copy the default zone file as a template:

```bash
sudo cp /etc/bind/db.127 /etc/bind/zones/anis.example.rev
```

Edit the reverse zone file:

```bash
sudo nano /etc/bind/zones/anis.example.rev
```

Replace the contents with:

```bind
$TTL    604800
@       IN      SOA     ns1.anis.example. root.ns1.anis.example. (
                              2026051701 ; Serial
                              604800     ; Refresh
                              86400      ; Retry
                              2419200    ; Expire
                              604800 )   ; Negative Cache TTL
;

@       IN      NS      ns1.anis.example.

; Reverse mappings
77      IN      PTR     ns1.anis.example.
60      IN      PTR     www.anis.example.
```

Check the reverse zone configuration:

```bash
sudo named-checkzone 12.17.172.in-addr.arpa /etc/bind/zones/anis.example.rev
```

## Step 7: Restart BIND Service

```bash
sudo systemctl restart bind9
```

## Step 8: Configure the DNS Server to Use Itself

Edit the `resolv.conf` file (or configure netplan):

```bash
sudo nano /etc/resolv.conf
```

Add or modify:

```bash
nameserver 172.17.12.77
search anis.example
```

> **Note:** On Ubuntu 22.04, `/etc/resolv.conf` is typically managed by systemd-resolved. For a permanent change, configure netplan (see Step 11).

## Step 9: Test DNS Resolution on the Server

Test forward lookup:

```bash
dig @localhost www.anis.example
```

Expected output (partial):

```bash
;; ANSWER SECTION:
www.anis.example.       604800  IN      A       172.17.12.60
```

Test reverse lookup:

```bash
dig -x 172.17.12.77 @localhost
```

Expected output (partial):

```bash
;; ANSWER SECTION:
77.12.17.172.in-addr.arpa. 604800 IN    PTR     ns1.anis.example.
```

Test ping:

```bash
ping ns1.anis.example
```

Expected output:

```bash
PING ns1.anis.example (172.17.12.77) 56(84) bytes of data.
64 bytes from ns1.anis.example (172.17.12.77): icmp_seq=1 ttl=64 time=0.057 ms
```

## Step 10: Configure Client Machines

On each client (`client1` and `client2`), edit the netplan configuration:

```bash
sudo nano /etc/netplan/00-installer-config.yaml
```

Example configuration:

```yaml
network:
  ethernets:
    ens160:
      addresses:
      - 172.17.12.60/24   # Use .54 for client2
      gateway4: 172.17.12.1
      nameservers:
        addresses:
        - 172.17.12.77
        search: [anis.example]
  version: 2
```

Apply the configuration:

```bash
sudo netplan try
```

If successful, apply it permanently:

```bash
sudo netplan apply
```

## Step 11: Test DNS Resolution on Clients

On both `client1` and `client2`, run the following commands:

```bash
nslookup ns1.anis.example
nslookup www.anis.example
nslookup 172.17.12.77
nslookup 172.17.12.60
```

Expected output format:

```bash
Server:         172.17.12.77
Address:        172.17.12.77#53

Name:   www.anis.example
Address: 172.17.12.60
```

Test connectivity between clients using names:

```bash
# From client1
ping client2.anis.example

# From client2
ping client1.anis.example
```

## Troubleshooting

### Check BIND logs

```bash
sudo journalctl -u bind9 -f
```

### Verify zone files after changes

```bash
sudo named-checkzone anis.example /etc/bind/zones/anis.example
sudo named-checkzone 12.17.172.in-addr.arpa /etc/bind/zones/anis.example.rev
```

### Test configuration without restarting

```bash
sudo named-checkconf
```

### Increment serial numbers

Whenever you make changes to zone files, **always increment the Serial number** (e.g., `2026051702`).

## Conclusion

You have successfully installed and configured a private BIND DNS server on Ubuntu 22.04. Your clients can now resolve internal hostnames and FQDNs without relying on external DNS servers. This setup provides:

- **Forward resolution** (names → IP addresses)
- **Reverse resolution** (IP addresses → names)
- **Recursive queries** for external domains via forwarders
- **Access control** using the trusted ACL

For production environments, consider adding a secondary DNS server, implementing DNSSEC, and setting up proper monitoring.
