# OAuth 2.0: The Complete Guide to Authorization

OAuth 2 lets third-party apps access user data without handling passwords. Instead of asking users for their credentials, OAuth 2 delegates authentication to the service provider (like Google or GitHub) and issues tokens that represent specific permissions.

Here's what you need to know: **OAuth 2 is an authorization framework, not authentication.** It answers "What can this app do?" not "Who is this user?" When you click "Sign in with Google" on a website, that's typically OpenID Connect (which adds authentication) built on top of OAuth 2.

## Why OAuth 2 Exists

Before OAuth 2, applications either asked users for passwords (security risk) or required users to share API keys (too permissive). OAuth 2 solves this by letting users grant limited, revocable permissions.

For example, a backup app can access your Google Drive files **without your Google password**, and you can revoke access anytime.

This guide shows you how OAuth 2 works, which grant types to use for different scenarios, and how to implement secure OAuth 2 flows in your applications.

---

## Key Takeaways

Before diving into the technical details, here are the essential points about OAuth 2:

- **OAuth 2 provides authorization, not authentication.** It allows applications to obtain permission to access resources but does not verify the user's identity on its own.

- **The OAuth 2 flow is defined by four core roles:** Resource Owner, Client, Authorization Server, and Resource Server, each of which plays a part in enabling secure access.

- **The Authorization Code grant** is the standard approach for server-side applications where the client secret can be kept confidential.

- **PKCE** (Proof Key for Code Exchange) is essential for public clients as it protects against authorization code interception attacks.

- **Tokens issued by OAuth 2 have limited lifespans:** access tokens expire, and refresh tokens allow applications to obtain new access tokens without user intervention.

- **Scopes in OAuth 2 define the permissions** being requested. Each authorization request specifies which resources and actions the application can access, such as read, write, or administrative privileges.

- **OAuth 1 is now deprecated.** OAuth 2 has replaced OAuth 1.0 due to improved security and a more straightforward implementation process.

## What is OAuth 2?

OAuth 2 is an authorization framework defined in **RFC 6749** that enables third-party applications to obtain limited access to a user's resources on an HTTP service. The framework standardizes how applications request and receive permission to access user data without exposing user credentials.

### The Valet Key Analogy

Think of OAuth 2 like a valet key for your car: you give a parking attendant a limited-use key that only opens the door and starts the engine, but doesn't open the trunk or glove compartment.

Similarly, OAuth 2 gives applications specific permissions (scopes) to access certain resources, without granting full account access.

### The Core Problem OAuth 2 Solves

Applications no longer need to store user passwords. Instead, users authorize applications through the service provider (like Google or GitHub), which issues tokens that represent the granted permissions. This reduces the attack surface and gives users granular control over what each application can access.

## How OAuth 2 Works: The Basic Flow

OAuth 2 follows a six-step pattern regardless of which grant type you use. Understanding this flow helps you debug issues and choose the right grant type for your application.

```
+----------+           +---------------+           +---------------+
|          |           |               |           |               |
|  Client  |           | Authorization |           | Resource      |
|  (App)   |           | Server        |           | Server (API)  |
|          |           |               |           |               |
+----+-----+           +-------+-------+           +-------+-------+
     |                         |                           |
     | 1. Authorization Request|                           |
     |------------------------>|                           |
     |                         |                           |
     | 2. User Authorization   |                           |
     |<----------------------->|                           |
     |                         |                           |
     | 3. Authorization Grant  |                           |
     |<------------------------|                           |
     |                         |                           |
     | 4. Access Token Request |                           |
     |------------------------>|                           |
     |                         |                           |
     | 5. Access Token &       |                           |
     |    Refresh Token        |                           |
     |<------------------------|                           |
     |                         |                           |
     | 6. API Request with     |                           |
     |    Access Token         |                           |
     |---------------------------------------------------->|
     |                         |                           |
     | 7. Protected Resource   |                           |
     |<----------------------------------------------------|
     |                         |                           |
```

### Step-by-Step Breakdown

| Step | Description |
|------|-------------|
| **Step 1** | **Application requests authorization.** Your application redirects the user to the authorization server with a request for specific permissions (scopes). |
| **Step 2** | **User authorizes the application.** The user logs in (if needed) and approves or denies your application's request. If approved, the authorization server generates an authorization grant. |
| **Step 3** | **Application requests access token.** Your application exchanges the authorization grant (plus your client credentials) for an access token. This happens server-to-server. |
| **Step 4** | **Authorization server issues access token.** If everything checks out, the server returns an access token (and optionally a refresh token) to your application. |
| **Step 5** | **Application requests protected resource.** Your application uses the access token to make API requests on behalf of the user. |
| **Step 6** | **Resource server serves the resource.** The API validates the token and returns the requested data if the token is valid and has the required scopes. |

> **Note:** The specific grant type you choose determines how steps 1-4 work, but steps 5-6 are the same for all flows.

## OAuth 2 vs Authentication vs OpenID Connect

Understanding the difference between OAuth 2, authentication, and OpenID Connect prevents common implementation mistakes.

| Aspect | OAuth 2 | Authentication | OpenID Connect |
|--------|---------|----------------|-----------------|
| **Purpose** | Authorization (permissions) | Identity verification | Both authentication and authorization |
| **Answers** | "What can this app do?" | "Who is this user?" | "Who is this user and what can they do?" |
| **Token Type** | Access token | Session token/cookie | ID token + access token |
| **Use Case** | API access, third-party integrations | Login systems | Social login, single sign-on |
| **Example** | App accessing your GitHub repos | Logging into your email | "Sign in with Google" button |

### When to Use Each

- **Use OAuth 2 alone** when you only need to grant API access (e.g., a backup tool accessing your cloud storage).

- **Use authentication** when building your own login system (e.g., username/password for your application).

- **Use OpenID Connect** when you need both identity and permissions (e.g., social login that also accesses user profile data).

## OAuth 2 Roles Explained

OAuth 2 defines four roles that interact during the authorization process. Understanding these roles clarifies how OAuth 2 works in practice.

```
+------------------+     +------------------+
|                  |     |                  |
|  Resource Owner  |     |   Client         |
|  (The User)      |     |   (Application)  |
|                  |     |                  |
+--------+---------+     +--------+---------+
         |                        |
         |   Grants Permission    |   Registers with
         |   to Access Resources  |   Client ID/Secret
         |                        |
         v                        v
+------------------+     +------------------+
|                  |     |                  |
| Authorization    |<----| Resource         |
| Server           |     | Server           |
| (Issues Tokens)  |     | (Hosts APIs)     |
|                  |     |                  |
+------------------+     +------------------+
```

### The Four Roles

| Role | Description | Real-World Example |
|------|-------------|-------------------|
| **Resource Owner** | The user who owns the resources and can grant access to them | You, when you click "Allow" to let an app access your Google Drive files |
| **Client** | The application requesting access to the user's resources | A mobile app, web application, or server-side service |
| **Authorization Server** | Authenticates the resource owner and issues access tokens | Google's authorization server for Google accounts |
| **Resource Server** | Hosts the protected resources and validates access tokens | DigitalOcean's API that provides droplet data |

> **Note:** From an application developer's perspective, the authorization server and resource server are often combined into a single "API" or "Service" that you're integrating with.

## Application Registration and Client Credentials

Before your application can use OAuth 2, you must register it with the service provider. Registration happens through the service's developer portal or API management interface.

### Application Registration Requirements

During registration, you'll provide:

| Field | Description |
|-------|-------------|
| **Application Name** | A human-readable name that users will see during authorization (e.g., "My Backup Tool") |
| **Application Website** | Your application's homepage or documentation URL |
| **Redirect URI (Callback URL)** | The exact URL where the authorization server will send users after they authorize or deny your application. This must match exactly what you specify in authorization requests |

### Client ID and Client Secret

After registration, the service issues client credentials:

| Credential | Description | Security Level |
|------------|-------------|----------------|
| **Client ID** | A publicly exposed string that identifies your application. Safe to include in client-side code, URLs, and public repositories. | Public |
| **Client Secret** | A confidential string that authenticates your application to the authorization server. Must be stored securely on your server. | Confidential |

### Security Critical: Never Expose Client Secrets

❌ **Do NOT expose client secrets in:**
- Frontend JavaScript code or mobile apps (use PKCE instead)
- Git repositories, even in `.env` files (use `.gitignore`)
- URLs or query parameters
- Client-side storage (localStorage, cookies)
- Error messages or logs
- Public documentation or screenshots

> ⚠️ **Warning:** Compromised secrets require regeneration and redeployment of all clients.

### Where to Store Client Secrets

✅ **Acceptable storage locations:**
- Environment variables
- Secret management services (AWS Secrets Manager, HashiCorp Vault)
- Encrypted configuration files
- Server-side only databases

## OAuth 2 Grant Types Overview

OAuth 2 defines four main grant types (flows) for different application scenarios:

| Grant Type | Best For | Security Level | Recommended |
|------------|----------|----------------|-------------|
| Authorization Code | Server-side web apps | High | ✅ Yes (with PKCE) |
| Implicit Flow | Single-page apps (legacy) | Low | ❌ Deprecated |
| Resource Owner Password | Trusted first-party apps | Medium | ⚠️ Limited use |
| Client Credentials | Machine-to-machine (M2M) | High | ✅ Yes |

---

## Grant Type 1: Authorization Code Flow

The Authorization Code flow is the most secure and commonly used OAuth 2 grant type. It's designed for server-side applications that can keep a client secret confidential.

### When to Use

- Server-side web applications (Node.js, Python, Ruby, Java, PHP, etc.)
- Applications that can securely store a client secret
- Any scenario where you need the highest level of security

### Why It's Secure

The authorization code is exchanged for an access token via a secure backchannel (server-to-server), preventing the token from being exposed to the user's browser.

### Flow Diagram

```
+----------+                               +---------------+
|          | ---(A)--- Authorization Request --->|               |
|          |     (redirects user)               |               |
|          |                               |  Authorization |
|  Client  | <---(B)--- Authorization Code ---|     Server     |
|  (Web    |     (via redirect)               |               |
|  Server) |                               |               |
|          | ---(C)--- Access Token Request--->|               |
|          |     (with client secret)          |               |
|          |                               |               |
|          | <---(D)--- Access Token --------|               |
+----------+                               +---------------+
     |
     | ---(E)--- API Request with Access Token ---> +---------------+
     |                                               |   Resource    |
     | <---(F)--- Protected Resource ---------------|     Server    |
     |                                               +---------------+
```

### Step-by-Step Implementation

**Step 1: Build the Authorization URL**

```
https://authorization-server.com/auth?
  response_type=code
  &client_id=YOUR_CLIENT_ID
  &redirect_uri=https://yourapp.com/callback
  &scope=read write
  &state=random_string_12345
```

**Step 2: Exchange Code for Token (Server-Side Request)**

```http
POST https://authorization-server.com/token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
&code=AUTHORIZATION_CODE
&redirect_uri=https://yourapp.com/callback
&client_id=YOUR_CLIENT_ID
&client_secret=YOUR_CLIENT_SECRET
```

**Step 3: Token Response**

```json
{
  "access_token": "eyJhbGciOiJSUzI1NiIsImtpZCI6...",
  "token_type": "Bearer",
  "expires_in": 3600,
  "refresh_token": "tGzv3JOkF0XG5Qx2TlKWIA",
  "scope": "read write"
}
```

**Step 4: Use Access Token for API Calls**

```http
GET https://api.example.com/user/profile
Authorization: Bearer eyJhbGciOiJSUzI1NiIsImtpZCI6...
```

### PKCE (Proof Key for Code Exchange)

For public clients (mobile apps, single-page apps) that cannot store a client secret securely, use **PKCE** (pronounced "pixie").

PKCE adds two dynamically generated values:
- `code_verifier`: A random string (43-128 characters)
- `code_challenge`: A SHA256 hash of the code_verifier

**Authorization Request with PKCE:**

```
https://authorization-server.com/auth?
  response_type=code
  &client_id=YOUR_CLIENT_ID
  &redirect_uri=https://yourapp.com/callback
  &code_challenge=H2lQ2FsLmUQzK5Z5xZ5xZ5xZ5xZ5xZ5xZ5xZ5xZw
  &code_challenge_method=S256
  &state=random_string_12345
```

**Token Request with PKCE:**

```http
POST https://authorization-server.com/token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
&code=AUTHORIZATION_CODE
&redirect_uri=https://yourapp.com/callback
&client_id=YOUR_CLIENT_ID
&code_verifier=dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk
```

## Grant Type 2: Implicit Flow (Deprecated)

The Implicit flow was designed for browser-based single-page applications (SPAs) to get access tokens without an intermediate code exchange step.

### Why It's Deprecated

- **No client authentication** - A malicious app can pretend to be the client
- **Credentials visible in source code** - Anyone can inspect the page and obtain credentials
- **Vulnerable to phishing attacks** - No secure backchannel
- **Access token exposed in URL fragment** - Can be intercepted

### Historical Flow (Do Not Use)

```
https://example-app.com/redirect#access_token=g0ZGZmPj4nOWIlTTk3Pw1Tk4ZTKyZGI3
  &token_type=Bearer
  &expires_in=400
  &state=xcoVv98y3kd55vuzwwe3kcq
```

### Modern Alternative

Use **Authorization Code Flow with PKCE** instead of the Implicit flow.

## Grant Type 3: Resource Owner Password Credentials Flow

In this flow, the user's username and password are exchanged directly for an access token. The user gives the app their credentials directly, and the app uses those credentials to get an access token.

### Flow Diagram

```
+----------+
|          |
|  Client  |
|  (App)   |
|          |
+----+-----+
     |
     | 1. User provides username/password
     |
     v
+----------+     +---------------+
|          |     |               |
|  Client  |---->| Authorization |
|  (App)   |     | Server        |
|          |<----|               |
+----------+     +---------------+
     |
     | 2. Access token returned
     |
     v
+----------+     +---------------+
|          |     |               |
|  Client  |---->| Resource      |
|  (App)   |     | Server        |
|          |<----|               |
+----------+     +---------------+
```

### Token Request

```http
POST https://authorization-server.com/token
Content-Type: application/x-www-form-urlencoded

grant_type=password
&username=johndoe
&password=P@ssw0rd123
&client_id=YOUR_CLIENT_ID
&client_secret=YOUR_CLIENT_SECRET
```

### When to Use (Very Limited)

- **Only for first-party applications** (apps you own and control)
- Legacy systems that cannot support modern OAuth flows
- Internal enterprise applications

### Why This Flow is NOT Recommended

| Risk | Description |
|------|-------------|
| **Impersonation** | No way to verify that the resource owner actually made the request |
| **Phishing Attacks** | Malicious apps can ask for credentials by pretending to be legitimate services |
| **Credential Leakage** | User credentials could be leaked maliciously to an attacker |
| **Over-permissioning** | A client app can request any scope without explicit user permission |

### Real-World Example of Abuse (2017)

In 2017, a fake Google Docs application was used to fool users into thinking it was a legitimate product offered by Google. The attackers used this app to access users' email accounts by abusing the OAuth token.

## Grant Type 4: Client Credentials Flow

The Client Credentials flow permits a client service to use its own credentials, instead of impersonating a user, to access protected data. This is for **machine-to-machine (M2M)** communication.

### When to Use

- Server-to-server API calls
- Background services and daemons
- Cron jobs that need to access APIs
- No user interaction required

### Flow Diagram

```
+----------+
|          |
|  Client  |
|  (Service)|
|          |
+----+-----+
     |
     | 1. Token Request with Client Credentials
     |
     v
+----------+     +---------------+
|          |     |               |
|  Client  |---->| Authorization |
|  (Service)|     | Server        |
|          |<----|               |
+----------+     +---------------+
     |
     | 2. Access Token returned
     |
     v
+----------+     +---------------+
|          |     |               |
|  Client  |---->| Resource      |
|  (Service)|     | Server        |
|          |<----|               |
+----------+     +---------------+
```

### Token Request

```http
POST https://authorization-server.com/token
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials
&client_id=YOUR_CLIENT_ID
&client_secret=YOUR_CLIENT_SECRET
&scope=read only
```

### Token Response

```json
{
  "access_token": "eyJhbGciOiJSUzI1NiIsImtpZCI6...",
  "token_type": "Bearer",
  "expires_in": 3600,
  "scope": "read only"
}
```

### Example: GitHub App Installation Token

```http
POST https://api.github.com/app/installations/12345678/access_tokens
Authorization: Bearer YOUR_JWT
Accept: application/vnd.github.v3+json
```

## Refresh Tokens

Refresh tokens allow applications to obtain new access tokens without requiring user interaction. This provides a better user experience while maintaining security.

### Token Lifespans

| Token Type | Typical Lifespan | Purpose |
|------------|------------------|---------|
| **Access Token** | 5-60 minutes | Short-lived, used for API calls |
| **Refresh Token** | Days to months | Long-lived, used to get new access tokens |

### Refresh Token Flow

```http
POST https://authorization-server.com/token
Content-Type: application/x-www-form-urlencoded

grant_type=refresh_token
&refresh_token=YOUR_REFRESH_TOKEN
&client_id=YOUR_CLIENT_ID
&client_secret=YOUR_CLIENT_SECRET
```

### Response

```json
{
  "access_token": "eyJhbGciOiJSUzI1NiIsImtpZCI6...",
  "token_type": "Bearer",
  "expires_in": 3600,
  "refresh_token": "NEW_REFRESH_TOKEN_IF_ROTATION"
}
```

### Security Best Practices for Refresh Tokens

- **Rotate refresh tokens** on each use
- **Store refresh tokens securely** (server-side only)
- **Implement revocation** for logout and suspicious activity
- **Set reasonable expiration** (hours for high-security apps, days/weeks for lower risk)

## Scopes and Permissions

Scopes define the specific permissions being requested. Each authorization request specifies which resources and actions the application can access.

### Common Scope Examples

| Service | Scope | Permission |
|---------|-------|------------|
| **GitHub** | `repo` | Full control of private repositories |
| **GitHub** | `user:email` | Read access to user's email addresses |
| **Google** | `https://www.googleapis.com/auth/drive.readonly` | View Google Drive files |
| **Google** | `https://www.googleapis.com/auth/gmail.send` | Send emails on your behalf |
| **Spotify** | `user-read-playback-state` | Read user's playback state |
| **Spotify** | `user-modify-playback-state` | Modify user's playback state |

### Scope Best Practices

1. **Request minimal scopes** - Only ask for what you need
2. **Use granular scopes** - Prefer `read` over `write` when possible
3. **Explain scope usage** - Tell users why you need each permission
4. **Handle scope changes** - Gracefully handle when users deny specific scopes

## Security Best Practices

### 1. Always Use HTTPS

```
✅ https://yourapp.com/callback
❌ http://yourapp.com/callback
```

### 2. Validate Redirect URIs

Register exact redirect URIs. Never allow open redirects or pattern matching:

```
✅ https://yourapp.com/oauth/callback
✅ https://yourapp.com/oauth/callback/v2
❌ https://yourapp.com/anything/goes
```

### 3. Use State Parameters

The `state` parameter prevents CSRF attacks. Generate a random string for each request.

```javascript
// Generate a secure random state
const state = crypto.randomBytes(32).toString('hex');

// Store it in a session or temporary storage
session.oauthState = state;

// Verify when the user returns
if (request.query.state !== session.oauthState) {
    throw new Error('CSRF attack detected');
}
```

### 4. Implement Token Revocation

Provide users a way to revoke application access:

```http
POST https://authorization-server.com/revoke
Content-Type: application/x-www-form-urlencoded

token=YOUR_TOKEN
&token_type_hint=access_token
&client_id=YOUR_CLIENT_ID
&client_secret=YOUR_CLIENT_SECRET
```

### 5. Rotate Secrets Regularly

- Rotate client secrets every 90-180 days
- Use environment variables or secret managers
- Never hardcode secrets in source code

### 6. Validate Tokens Properly

```javascript
function validateAccessToken(token, requiredScopes) {
    try {
        const decoded = jwt.verify(token, publicKey, {
            algorithms: ['RS256'],
            issuer: 'https://auth.example.com',
            audience: 'https://api.example.com'
        });
        
        // Check expiration
        if (decoded.exp < Date.now() / 1000) {
            throw new Error('Token expired');
        }
        
        // Check required scopes
        const hasScopes = requiredScopes.every(scope => 
            decoded.scope && decoded.scope.includes(scope)
        );
        
        if (!hasScopes) {
            throw new Error('Insufficient scopes');
        }
        
        return decoded;
    } catch (error) {
        return null;
    }
}
```

## Comparison Table: Which Grant Type to Use?

| Scenario | Grant Type | Why |
|----------|------------|-----|
| Web app with backend server | Authorization Code with secret | Can securely store client secret |
| Mobile app (iOS/Android) | Authorization Code with PKCE | Cannot store client secret securely |
| Single-page app (React/Vue) | Authorization Code with PKCE | Modern alternative to deprecated Implicit flow |
| Server-to-server API calls | Client Credentials | No user involved |
| First-party trusted app | Resource Owner Password | Only for legacy or internal use |
| Sign in with provider | OpenID Connect (built on OAuth 2) | Need user identity, not just permissions |

## Common OAuth 2 Vulnerabilities and Mitigations

| Vulnerability | Description | Mitigation |
|---------------|-------------|------------|
| **CSRF** | Attacker tricks user into authorizing malicious app | Use `state` parameter |
| **Redirect URI manipulation** | Attacker intercepts code via open redirect | Strict URI validation, exact matching |
| **Code interception** | Attacker steals authorization code | Use PKCE for public clients |
| **Token leakage** | Token exposed in logs or URLs | Use HTTPS, never log tokens |
| **Weak token validation** | App doesn't properly validate tokens | Validate signature, issuer, audience, expiry |
| **Refresh token abuse** | Stolen refresh token used to get new tokens | Implement token rotation, short lifetimes |

## Summary

OAuth 2 is the industry standard for authorization, enabling secure, delegated access to user resources. Key points to remember:

1. **OAuth 2 is for authorization**, not authentication (use OpenID Connect for identity)

2. **Choose the right grant type** based on your application type and security requirements

3. **Authorization Code flow with PKCE** is the recommended approach for most applications

4. **Never expose client secrets** in client-side code

5. **Use short-lived access tokens** with refresh tokens for long-lived access

6. **Always validate tokens** and use HTTPS everywhere

### Quick Decision Tree

```
Is there a user involved?
├─ NO → Use Client Credentials flow
└─ YES → Can you securely store a client secret?
    ├─ YES (server-side web app) → Use Authorization Code flow
    └─ NO (mobile/SPA) → Use Authorization Code flow with PKCE
```
