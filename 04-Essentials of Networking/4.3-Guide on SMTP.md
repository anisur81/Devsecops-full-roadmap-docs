
# Brief Discussion on Simple Mail Transfer Protocol (SMTP)

## Introduction

SMTP is an application-layer protocol used to send and transfer email between servers. It relies on a TCP connection to port 25 to reliably deliver messages from a client to an email server.

- Uses TCP for reliable message transmission
- Servers listen on port 25 for incoming email
- Client establishes a TCP connection before sending mail
- Core protocol for email delivery between mail servers

> **Important:** SMTP does NOT retrieve emails. Retrieval protocols include:
> - POP3
> - IMAP

## Types of SMTP Protocol

The SMTP model supports two types of email delivery methods: **end-to-end** and **store-and-forward**.

| Type | Description | Use Case |
|------|-------------|----------|
| **End-to-end** | Email is sent directly from sender's SMTP client to recipient's SMTP server without passing through intermediate servers | Between organizations |
| **Store-and-forward** | Email may pass through several intermediate servers (MTAs) before reaching the recipient | Within organizations with TCP/IP and SMTP-based networks |

## Model of SMTP System

![SMTP System Model](media/image2.png)

| Component | Description |
|-----------|-------------|
| **User at a Terminal (Sender)** | The person composing the email using a mail application (e.g., Outlook, Thunderbird, webmail) |
| **User Agent (Sender)** | The email client software that allows the sender to create, edit, and send emails. Hands outgoing email to the system's mail queue |
| **Queue of Mail to Be Sent** | A temporary storage area where outgoing emails wait before being processed by the mail server (MTA) |
| **Message Transfer Agent (Client Side)** | Software responsible for sending emails across the network; retrieves messages from outgoing queue and communicates with receiving server |
| **TCP Connection (Port 25)** | Client's MTA opens a TCP connection to server's MTA on port 25; SMTP commands, responses, and mail data are exchanged |
| **Message Transfer Agent (Server Side)** | Receiving email server accepts incoming SMTP connection, processes incoming email, and stores it for the user |
| **User Mailboxes** | Server stores received messages in individual user mailboxes |
| **User Agent (Receiver)** | Email application used by recipient to access the mailbox; retrieves emails using POP3 or IMAP |
| **User at a Terminal (Receiver)** | Person reading the received email on their device |

## Components of SMTP

| Component | Full Form | Description |
|-----------|-----------|-------------|
| **MUA** | Mail User Agent | Computer application that helps in sending and retrieving mail; responsible for creating email messages for transfer to the MTA |
| **MSA** | Mail Submission Agent | Computer program that receives mail from MUA and interacts with MTA for mail transfer |
| **MTA** | Mail Transfer Agent | Software that transfers mail from one system to another with the help of SMTP |
| **MDA** | Mail Delivery Agent | System that helps in delivery of mail to the local system (also known as Local Delivery Agent) |

## How Does SMTP Work?

![SMTP Workflow](media/image1.png)

### 1. Sending Email

- When a user wants to send an email, they use a User Agent (UA), like Outlook or Gmail
- The email is handed over to the MTA, which is responsible for transferring the email to the recipient's mail server

### 2. SMTP Client and Server

| Role | Description |
|------|-------------|
| **Sender-SMTP (Client)** | Email sender's MTA initiates the connection to the recipient's MTA |
| **Receiver-SMTP (Server)** | Receiving MTA listens for incoming connections and receives the email from sender-SMTP |

- This communication happens over TCP port 25

### 3. Relays and Gateways

| Component | Description |
|-----------|-------------|
| **Relays** | Intermediate MTAs that email may pass through before reaching destination server |
| **Gateways** | Converts email to appropriate format if sending and receiving systems use different email protocols |

### 4. Email Delivery

1. Sender's MTA sends email to receiver's MTA (directly or through relays)
2. MTA uses SMTP protocol to transfer the message
3. Once delivered to destination MTA, email is placed in recipient's mailbox
4. Recipient's User Agent (UA) can then download the email

## SMTP Envelope

### Purpose

- Contains information that guides email delivery between servers
- Distinct from email headers and body
- Not visible to the email recipient

### Contents

| Component | Description |
|-----------|-------------|
| **Sender Address** | Specifies where the email originates |
| **Recipient Addresses** | Indicates where the email should be delivered |
| **Routing Information** | Helps servers determine the path for email delivery |

### Comparison to Regular Mail

Think of the SMTP envelope as the **address on a physical envelope** for regular mail. Just like an envelope guides postal delivery, the SMTP envelope directs email servers on where to send the email.

## SMTP Commands

| S.No. | Keyword | Command Form | Description | Usage |
|-------|---------|--------------|-------------|-------|
| 1 | HELO | `HELO<SP><domain><CRLF>` | Provides identification of the sender (host name) | Mandatory |
| 2 | MAIL | `MAIL<SP>FROM:<reverse-path><CRLF>` | Specifies the originator of the mail | Mandatory |
| 3 | RCPT | `RCPT<SP>TO:<forward-path><CRLF>` | Specifies the recipient of mail | Mandatory |
| 4 | DATA | `DATA<CRLF>` | Specifies the beginning of the mail | Mandatory |
| 5 | QUIT | `QUIT<CRLF>` | Closes the TCP connection | Mandatory |
| 6 | RSET | `RSET<CRLF>` | Aborts current mail transaction but keeps TCP connection open | Highly recommended |
| 7 | VRFY | `VRFY<SP><string><CRLF>` | Confirms or verifies the user name | Highly recommended |
| 8 | NOOP | `NOOP<CRLF>` | No operation | Highly recommended |
| 9 | TURN | `TURN<CRLF>` | Reverses the role of sender and receiver | Seldom used |
| 10 | EXPN | `EXPN<SP><string><CRLF>` | Specifies the mailing list to be expanded | Seldom used |
| 11 | HELP | `HELP<SP><string><CRLF>` | Sends specific documentation to the system | Seldom used |
| 12 | SEND | `SEND<SP>FROM:<reverse-path><CRLF>` | Sends mail to the terminal | Seldom used |
| 13 | SOML | `SOML<SP>FROM:<reverse-path><CRLF>` | Sends mail to terminal if possible; otherwise to mailbox | Seldom used |
| 14 | SAML | `SAML<SP>FROM:<reverse-path><CRLF>` | Sends mail to terminal and mailbox | Seldom used |

## SMTP Ports

![SMTP Ports](media/image4.png)

| Port | Protocol | Use Case | Status |
|------|----------|----------|--------|
| **587** | SMTP with TLS | Secure SMTP submission for client-to-server communication | **Recommended** (most commonly used) |
| **465** | SMTPS | Secure SMTP | No longer official standard; not recommended |
| **25** | SMTP | SMTP relay between mail servers | Often blocked by ISPs for outgoing mail |
| **2525** | SMTP (alternative) | Alternative for SMTP submission | Not official; used when port 25 is blocked |

## Advantages of SMTP

- If necessary, users can have a dedicated server
- Allows for bulk mailing
- Low cost and wide coverage area
- Offers choices for email tracking
- Reliable and prompt email delivery

---

## Disadvantages of SMTP

- SMTP's common port can be blocked by several firewalls
- SMTP security is a bigger problem
- Its simplicity restricts how useful it can be
- Only 7-bit ASCII characters can be used
- If a message is longer than a certain length, SMTP servers may reject the entire message
- Delivering your message will typically involve additional back-and-forth processing between servers, which will delay sending and raise the likelihood that it won't be sent
